## Executive summary

Three findings against V12.2 net-new code: **0 CRITICAL, 1 HIGH, 1 MEDIUM, 1 INFO.** All findings are availability or economic-DoS shapes, not pool-drain primitives. Concretely:

- **R7-A MEDIUM** — Min-premium cancel cycle leaks ~3.8 ADA from the LP per round-trip on small-policy (`premium = 2 ADA`) Underwrite + Cancel sequences. Bounded by tx-fee economics, not a free drain; recommend raising `min_premium` to a value where `raw_fee_cancel ≥ min_utxo`.
- **R7-B HIGH (availability)** — `policy.ak::batch_oracles_uniform` non-exhaustive `when (sent_provider, this_provider) is { ... }` switch missing the `(Indigo, Indigo)` arm. Two-policy BatchClaim of Indigo policies is unconditionally rejected by the validator. Single-policy Claim and single-Indigo BatchClaim both work (the `seen` sentinel short-circuits the equality check on the first element). This is a functionality gap, not a security drain — no value is at risk; users with multiple Indigo-bound policies cannot batch their claims and must fall back to single-policy `Claim`.
- **R7-C INFO** — Off-chain `policy.ak::Cancel`'s `expected_pool_amount` check uses `>=` while `pool.ak::AcceptCancellation` enforces `==` on the new pool continuation. The `>=` is intentionally permissive (the pool's own enforcement is the binding leg), but the asymmetry is worth documenting in the auditor handoff so the spec doc's value-flow narrative isn't read as a tight cross-validator constraint.

All 13 prior round 1-6 findings still closed (regression sweep below). The 6 new Aiken tests pin R7-A and R7-B numerically + lock the disjointness + cross-pair invariants.

**Recommendation:** **Merge-with-follow-up.** R7-B is a single-line fix; R7-A is a min_premium-policy tuning decision; R7-C is a docs change. No code path identified that drains the pool, mints unauthorized tokens, or bypasses oracle pins. The auditor handoff is structurally ready.

## NEW V12.2 findings

### R7-A [MEDIUM] — Min-premium cancel cycle drains LP at ~3.8 ADA per round-trip

**Severity:** MEDIUM (economic DoS; not free money, bounded by tx fees, but LP-loss:attacker-cost ratio is ≥ 4×).

**Attack vector.** An attacker initiates `Underwrite { coverage: any_positive_value, premium: 2_000_000 }` (= `min_premium`) against a V12.2 pool with `protocol_fee_bps = 200`. Inside the 1-hour cancellation window the same insured submits `Cancel` + `AcceptCancellation`.

**Why the current code admits it.** The V12.2 Hybrid fee model moves the `min_utxo` floor INSIDE the carve on both the Underwrite path and the AcceptCancellation path.

- At Underwrite: `pricing.ak::calculate_fee_total(2_000_000, 200, 2_000_000) = max(2_000_000, 40_000) = 2_000_000`. So `net_pool_growth = 2_000_000 - 2_000_000 = 0`. Pool's `total_liquidity` and lovelace are both unchanged. Team gets 2 ADA via the required team output (funded from submitter wallet inputs).
- At Cancel: `cancellation_fee = premium − calculate_refund(premium) = 2_000_000 − 1_800_000 = 200_000` (= 0.2 ADA). `pricing.ak::calculate_fee_total(200_000, 200, 2_000_000) = max(2_000_000, 4_000) = 2_000_000`. The validator's value invariant `pool.ak:1264` (`cont == own − refund − fee_total_cancel`) extracts `1_800_000 + 2_000_000 = 3_800_000` lovelace from the pool. Team gets another 2 ADA, insured gets 1.8 ADA refund.

Net pool delta per cycle: `+0 − 3_800_000 = −3_800_000` lovelace (LP loses 3.8 ADA). Attacker burn (excl. tx fees): `premium − refund = 200_000` lovelace = 0.2 ADA. Loss-to-cost ratio: **19× on the bare math, ~4–8× including realistic tx fees on preprod (~0.5–1 ADA per tx, two txs per cycle).**

**Code location.** `validators/pool.ak:1242-1264`. The fee-total-on-cancel computation does not check that `cancellation_fee >= min_utxo / fee_bps × 10_000` (= 100 ADA premium at the default 2% rate). The V12.2 spec §3.6 explicitly describes the 100-ADA case (cancellation_fee = 10 ADA, raw_fee_cancel = 0.2 ADA, fee_total_cancel = 2 ADA) but does NOT discuss the min-premium edge case where `premium = 2 ADA`. The economic asymmetry on a 2-ADA cancel is structural to Hybrid fee + B2 cancel-split.

**Concrete exploit tx.**
- Underwrite: `redeemer = Underwrite { coverage: 100_000_000, premium: 2_000_000 }`, single policy output at `policy_validator` with `lovelace = 100_000_000`, `PolicyDatum.premium_paid = 2_000_000`, `oracle_provider = AegisSelf`, `oracle_nft = <ADA AegisSelf>`. Team output ≥ 2 ADA at `team_address`. Treasury donation = 1_000 lovelace (=`2_000_000 × 200 × 2_500 / 1e8`). User pays: 100 + 2 + 0.001 + tx_fee ≈ 102.5 ADA out of pocket.
- Wait < cancellation_window (1h) but not ITM (oracle still > strike).
- Cancel + AcceptCancellation: `redeemer = AcceptCancellation`, `PolicyRedeemer = Cancel`, signed by insured. Insured output ≥ 1.8 ADA. Pool continuation lovelace = `own − 3.8 ADA`. Team output ≥ 2 ADA. Treasury donation = 1 lovelace (cancel-path). Submitter receives ~100 ADA back (the policy's coverage, less tx fees).

After cycle: attacker burned `0.2 ADA premium retained + ~2 × 0.3 ADA tx fees + ~1 lovelace treasury donations = ~0.8 ADA`. LP lost 3.8 ADA. Team gained 4 ADA (≈ operator).

**Recommended fix.** Three options, ranked by simplicity:

1. **Raise `min_premium` to `min_utxo × 10_000 / cancellation_fee_bps / protocol_fee_bps = 100 ADA`** (the threshold where `raw_fee_cancel >= min_utxo`). Any cancel-path fee_total then equals the raw cancellation_fee × protocol_fee_bps directly with no floor uplift. Pool gains `net_pool_growth ≥ 0` at every Underwrite + cancel-path is value-neutral or positive. **Trade-off:** raises the minimum useable premium for users — UX impact, but matches the V12.2 §10.1 stated invariant that LP absorption is bounded.
2. **Apply the floor on cancel only when `cancellation_fee >= min_utxo`** (skip the team output entirely on micro-cancels; treat as a "no fee" cancel). **Trade-off:** team gets 0 on micro-cancels; off-chain analytics see a discontinuity at the threshold.
3. **Accept the loss; document explicitly in §10.1 as the "min-premium cancel grief surface."** Operator chooses to keep `min_premium = 2 ADA` and accepts the bounded LP exposure (`< 4 ADA per policy × N policies`). At the operator's own throughput estimate (~10 policies/day mainnet), the LP loss is bounded at ~40 ADA/day. **Trade-off:** does nothing about the asymmetry; needs explicit operator acceptance in writing.

**Regression test added.** `redteam_round_7_r7_a_min_premium_cancel_costs_lp_3_8_ada` + `_cycle_net_pool_delta_negative` + `_attacker_burn_below_lp_loss`. All three pass under current code (locking the buggy state numerically). When the fix lands, the tests must be updated alongside.

### R7-B [HIGH (availability)] — `batch_oracles_uniform` rejects two-Indigo batch via non-exhaustive switch

**Severity:** HIGH — breaks a documented V12.2 feature (Indigo BatchClaim) but does not put value at risk. Availability bug, not a security drain. Single-policy `Claim` for Indigo policies still works.

**Attack vector / functionality break.** A user holds two Indigo-bound policies (e.g., two iUSD depeg covers) and submits a BatchClaim that consumes both. The validator rejects via `policy.ak::batch_oracles_uniform`. The user falls back to two separate `Claim` txs, paying 2× tx fees. **Concrete tx flow:**
- `inputs`: 2 policy UTxOs at `policy_validator`, both with `oracle_provider = Indigo`, both with `oracle_nft = <iUSD>`.
- `redeemer = BatchClaim`, single price feed in `reference_inputs`.
- `policy.ak:228-282` invokes `batch_oracles_uniform`. Inside (line 84):
  ```aiken
  let same_provider =
    when (sent_provider, this_provider) is {
      (Charli3, Charli3) -> True
      (Orcfax, Orcfax) -> True
      (AegisSelf, AegisSelf) -> True
      _ -> False
    }
  ```
- `sent_provider = Indigo, this_provider = Indigo` falls into `_ -> False`. `ok ← False`. Result `all_equal = False`. BatchClaim rejects.

**Why the current code admits it (code location).** `validators/policy.ak:84-90`. The `OracleProvider` enum was extended from 3 to 4 variants in V12.2 Wave 2.3 (types.ak:88, adds `Indigo`). The `batch_oracles_uniform` helper was not updated to mirror the new variant. The dispatcher in `lib/aegis/oracle.ak::resolve_oracle_price` (lines 80-89) DOES handle all 4 variants exhaustively because Aiken's `when` checker requires it for a non-tuple expression. The tuple-pattern in `batch_oracles_uniform` uses a wildcard `_ -> False` that bypasses the exhaustivity check, hiding the gap.

**Single-policy Indigo Claim is unaffected.** The fold's `seen` sentinel guard means the first input never runs the equality check. With only one Indigo input, `seen = False → (pair, ok, True)` on the first iteration and the fold ends with `ok = True`. So Indigo single-policy claims succeed.

**Concrete exploit / non-exploit.** Not an exploit — the attack surface is "user pays 2× tx fees because batch path is broken." No pool drain, no oracle bypass, no malformed value flow.

**Recommended fix.** One-line change in `validators/policy.ak:84-90`:
```aiken
let same_provider =
  when (sent_provider, this_provider) is {
    (Charli3, Charli3) -> True
    (Orcfax, Orcfax) -> True
    (AegisSelf, AegisSelf) -> True
    (Indigo, Indigo) -> True     // NEW
    _ -> False
  }
```
Or, more robust against future variant additions: rewrite as `sent_provider == this_provider` (relying on `OracleProvider` being trivially Eq once compiler upgrades to support it; currently it does for sum-types with no fields). The existing `_ -> False` cross-pair rejection is preserved by the equality check.

**Regression test added.** `redteam_round_7_r7_b_batch_oracles_uniform_rejects_indigo_pair` + `_accepts_aegis_self_pair`. The first test currently asserts the buggy behavior (`same_provider == False` for two Indigo policies) and pins the regression; the second test locks the AegisSelf same-pair acceptance so the fix doesn't accidentally regress that path. When the R7-B fix lands, the first test's assertion must flip to `== True` (and the test name should be renamed `_accepts_indigo_pair`).

### R7-C [INFO] — Cross-validator value-flow asymmetry on cancel

**Severity:** INFO. No exploit; documentation gap only.

**Observation.** The V12.2 spec §3.6 describes the cancel-path pool delta as `cont == own − refund_amount − fee_total_cancel`. The actual pool validator enforces this with `==` (`pool.ak:1264`). But the policy.Cancel validator (`policy.ak:393-395`) also gates the new pool continuation's lovelace via `output_has_min_lovelace(out, expected_pool_amount)` where `expected_pool_amount = policy_utxo_lovelace − refund_amount`. The policy-side check is `>=` (lower bound) — and is trivially satisfied because `policy_utxo_lovelace` is just the policy's coverage and the pool's continuation is much larger.

**Concrete asymmetry.** For a 100-ADA premium / 100-ADA coverage policy cancel:
- policy.Cancel: cont >= `100 − 90 = 10` ADA. Pool has e.g. 1000 ADA, so trivially passes.
- pool.AcceptCancellation: cont == `1000 − 90 − 2 = 908` ADA. Exact equality.

Both pass simultaneously because pool's `==` value is the binding leg and policy's `>=` is permissive. **No exploit.** The asymmetry is by design — pool.AcceptCancellation is the value-flow truth source.

**Doc gap.** The auditor handoff at `docs/communications/AUDITOR_NOTIFY_V12_2_2026-05-12.md` (V12.2 Wave 7 deliverable) should explicitly call out that the dual-check pattern is intentional and that the pool-side `==` is the binding constraint.

**Recommended fix.** Add a paragraph to V12.2 spec §3.6 explaining the dual-check asymmetry. No code change required.

**Regression test added.** `redteam_round_7_r7_c_orcfax_fsp_constants_still_in_tree` (defensive pin against an accidental cleanup-deleting the Orcfax FSP constants that types.ak intentionally keeps in tree for V12.2 §10.6 reactivation cost).

## Additional V12.2-specific defense-in-depth checks

These items were verified during the review but not classed as findings because they're either already correctly implemented or out-of-scope:

### Indigo NFT-spoof / address-mismatch / single-token-of-policy
- `lib/aegis/oracle/indigo.ak:188-217` (`find_feed_datum`) enforces all 3 layers: (1) `expect list.has(indigo_canonical_nfts, oracle_nft)`, (2) `expect Some(expected_script_hash) = resolve_paired_script_hash(oracle_nft)` + `at_indigo_script = Script(h) == expected_script_hash`, (3) `expect single_token_of_policy(output.value, oracle_nft)`. The paired-list helper (`do_resolve_paired`, `:105-126`) is index-position-based, NOT "any script from list × any NFT from list" (which would have been the vulnerability). Verified by reading and confirmed by tests `indigo_pair_resolve_maps_iusd_nft_to_mock_script_hash`, `_maps_ibtc_ieth_isol_to_same_mock`, `_returns_none_for_unknown_nft`, `_returns_none_for_empty_nft`.

### Reference-input collision (2+ Indigo UTxOs with same canonical NFT)
- `lib/aegis/oracle/indigo.ak:153-176` (`find_feed_output`) returns the FIRST matching UTxO. **Not a 2+-NFT problem** because the canonical NFT is one-shot-minted (operator on preprod, Indigo's parameterized one-shot mint on mainnet), so 2 UTxOs carrying the same NFT means quantity > 1 across the policy, which the `single_token_of_policy` check (`:138-143`) explicitly catches: `dict.foldr(tokens, 0, _ + qty) == 1`. So even if an attacker SPLITS the one-shot mint across two UTxOs (1 + 0 quantity each, with NFT replicated across them), the validator's check sees total = 1 across the first matching UTxO only and would still accept that one. **Defense-in-depth holds.**

### Soft-disabled Charli3 / Orcfax leak surface
- Audited all 6 call sites that touch `PolicyDatum.oracle_provider` (per V12.2 spec §10.2): Underwrite (uses `policy_output_matches_underwrite` → `is_canonical_oracle_nft` line 450), BatchUnderwrite (uses `batch_policies_match_totals_v12` → `is_canonical_oracle_nft` line 299), ProcessClaim/AcceptCancellation/BatchExpireProcess (consume already-gated policies; no second check), policy.ak Claim/BatchClaim/Cancel (delegate to `resolve_oracle_price` → `charli3.resolve` or `orcfax.resolve` → leading `fail`). **No path bypasses the soft-disable.**

### Variant-constructor index alignment
- Verified `OracleProvider` indices: Charli3=0, Orcfax=1, AegisSelf=2, Indigo=3 — matches `types.ak:60-89` declaration order and the off-chain `OracleProviderIndigo.CONSTR_ID = 3` in `api/policies.py`.

### `BatchFeeTotals` field drop (V12: 8 fields → V12.2: 7 fields)
- V12.2 collapses V12's two-tier floored/unfloored tracking into a single canonical "actual lovelace routed" sum set. Verified at `validators/pool.ak:122-131`. The dropped fields (`team_total_unfloored`, `partner_totals_unfloored_sum`) are replaced by the structural property that `team_total + partner_totals_sum == fee_total_sum` (helper contract) — so no information is lost. The pool's value invariant now reads `cont == own + (total_premium − fee_total_sum)` directly, with no separate "unfloored sum" tracking. **No regression.**

### Pre-flight asset normalize regression (operator-action item from Wave 6)
- The Wave 6 smoke surfaced a wiring oversight at `api/policies.py:1900` and `:3879` where `_aegis_self_normalize_asset(asset)` was called unconditionally BEFORE provider dispatch in `_resolve_oracle_nft_bytes`. This is fixed in branch HEAD `b323af7` (commit `fix(v12.2): provider-aware pre-flight asset normalize (Wave 6 follow-up)`). Off-chain only; on-chain validator unaffected. Verified by re-reading `api/policies.py` HEAD state.

## Recommendation

**Merge-with-follow-up.** No CRITICAL findings. The two actionable items are:

1. **R7-B fix (HIGH availability)** — single-line change in `validators/policy.ak:84-90`, deploy in v12.3. Until fixed, Indigo BatchClaim is a documented limitation (single-policy Claim works). Mention in user-facing FAQ as "Indigo policies must be claimed one at a time until v12.3."

2. **R7-A operator decision (MEDIUM economic DoS)** — choose between (a) raise `min_premium` to 100 ADA, (b) skip team output on micro-cancels, or (c) accept the bounded LP exposure with an explicit §10.1 note. **Recommend (a) for mainnet** — the user-visible UX impact is small (most policies are 100+ ADA premium by design intent) and it eliminates the asymmetry cleanly.

3. **R7-C doc gap (INFO)** — add a paragraph to V12.2 spec §3.6 explaining the dual-check asymmetry (policy.Cancel's `>=` vs pool.AcceptCancellation's `==`). Auditor handoff at `docs/communications/AUDITOR_NOTIFY_V12_2_2026-05-12.md` should mention this.

The auditor handoff is structurally ready. R7-A + R7-B can be folded into v12.3 alongside the deferred L-002 (Expire multi-policy aggregation) and ECON-1 (BatchClaim same-insured) work.
