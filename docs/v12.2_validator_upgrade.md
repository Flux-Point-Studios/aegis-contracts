## 1. Executive Summary

V12.2 ships four coordinated changes under a single audit pass and a single coordinated redeploy. All four are small in surface but each carries an economic or trust-anchor consequence, so the auditor reviews them as one unit.

**(1) Hybrid fee model.** V12 carves the protocol fee from the premium (`net_premium = premium − raw_fee`) but applies the 2-ADA min-utxo floor as a *separate* submitter-paid pad layered on top of the carve. So a 20-ADA premium produces a 0.4-ADA `team_cut`, padded to 2 ADA with a 1.6-ADA exogenous floor pad sourced from the submitter's wallet — the user's total outflow becomes `premium + treasury_cut + floor_pad + tx_fee`, with the floor pad varying by premium size. V12.2 moves the floor *INSIDE* the carve: `fee_total = max(min_utxo_lovelace, raw_fee)`, `net_pool_growth = premium − fee_total`. A 20-ADA premium produces `fee_total = 2 ADA`, `net_pool_growth = 18 ADA`, team gets 2 ADA, partner 0, treasury 0.10 ADA, user pays exactly `20.10 ADA` excl. tx_fee. No separate floor-pad line. User-out simplifies from `premium + treasury + floor_pad` to `premium + treasury`. The validator's pool invariant flips from `+ net_premium` to `+ (premium − fee_total)`.

**(2) Soft-disable Charli3 + Orcfax.** The two variants stay in the `OracleProvider` sum type so V11 policies whose datums still decode under V12.2-class validator code (cross-validator policy snooping helpers, off-chain analytics) can parse them. But the on-chain canonical-NFT predicate `is_canonical_oracle_nft` returns `False` for both — meaning no new policy on V12.2 can bind to Charli3 or Orcfax. The Charli3 resolver in `oracle/charli3.ak` and the Orcfax resolver in `oracle/orcfax.ak` stay in the tree as dead code (Aiken doesn't have `@deprecated` as of v1.1.21 — we add a docstring header marker + a leading `fail` in any callable entry point so an accidental import at runtime aborts). The point is zero runtime call path through them under V12.2.

**(3) Indigo as 4th OracleProvider — 4 iAssets directly bound.** The `OracleProvider` sum type grows from 3 variants (Charli3, Orcfax, AegisSelf) to 4 (adds `Indigo`). Indigo runs their own on-chain oracle (each iAsset has a parameterized oracle script at its own address, with a canonical NFT pinning the UTxO and a Plutus V2 `OracleDatum = Constr_0[Constr_0[price_int], expiration_ms]` carrying price in 1e6 scale + POSIX-ms expiry). We bind to four iAssets directly: **iUSD, iBTC, iETH, iSOL** — each with its own canonical NFT policy id and per-iAsset oracle script hash baked in `lib/aegis/types.ak`. AegisSelf's canonical NFT allowlist stays at 5 (ADA/BTC/ETH/USDC/USDT — no iUSD added). USDA + USDM stay in the depeg registry with `wired: false` and a "Coming soon · pending data source" pill. Indigo is mainnet-only — Wave 4 deploys 4 preprod-only mock UTxOs (operator-owned script address + one-shot mock NFTs) mirroring Indigo's mainnet datum shape so the dev loop and preprod soak can exercise the Indigo-bound paths without touching mainnet. The mainnet NFT policy ids + script hashes are filled in from the live data validated during Wave 1.5; preprod values are TBD until Wave 4 mints the mocks. See §2.6 for the on-chain architecture, §3.4a for the parser, §5b for the mock.

**(4) Silent partner_cut absorb.** When `partner_cut_raw = fee_total × partner_share_bps / 10_000` is below `min_utxo_lovelace`, the partner output is *not* emitted and the entire `fee_total` flows to `team_cut` instead. PolicyDatum still records `partner_address` and `partner_share_bps` for off-chain analytics (the partner registry can reconstruct "should have received" amounts). No on-chain bookkeeping for absorbed attribution. The team_cut just gets larger; team_address is compile-time pinned, so no value diverts and no attack surface. The fee math (validator side) becomes:

```
raw_fee         = premium × fee_bps / 10_000             # 200 bps = 2%
fee_total       = max(min_utxo_lovelace, raw_fee)         # 2 ADA floor INSIDE carve
partner_cut_raw = fee_total × partner_share_bps / 10_000  # bps of FEE, capped 2000
partner_cut     = partner_cut_raw if partner_cut_raw >= min_utxo_lovelace else 0
team_cut        = fee_total − partner_cut                 # absorbs sub-floor partner_cut
treasury_cut    = premium × protocol_fee_bps × treasury_share_bps / 10_000^2  # unchanged
```

Validator checks (per Underwrite):

```
cont_pool.lovelace == own.lovelace + (premium − fee_total)
output_to_address_at_least(outputs, team_address, team_cut)
partner_cut > 0 → output_to_address_at_least(outputs, partner_address, partner_cut)
body.treasury_donation >= treasury_cut
```

Submitter outflow: `premium + treasury_cut + tx_fee` (clean — no separate floor pad).

**Worked examples** (locked into Section 14 as the canonical economics table):

| Premium | Partner | partner_share_bps | raw_fee | fee_total | partner_cut_raw | partner_cut (paid) | team_cut | net_pool_growth | treasury | User pays (excl tx) |
|---|---|---|---|---|---|---|---|---|---|---|
| 20 ADA | None | 0 | 0.4 | 2 | 0 | 0 | 2 | 18 | 0.10 | 20.10 |
| 100 ADA | None | 0 | 2 | 2 | 0 | 0 | 2 | 98 | 0.50 | 100.50 |
| 500 ADA | partner_X | 1500 | 10 | 10 | 1.5 | 0 (absorbed) | 10 | 490 | 2.50 | 502.50 |
| 1000 ADA | partner_X | 1500 | 20 | 20 | 3 | 3 | 17 | 980 | 5 | 1005 |

The 500-ADA / partner_X row is the canonical *silent absorb* case: `partner_cut_raw = 1.5 ADA < 2 ADA` → partner output omitted, `team_cut = fee_total = 10 ADA`. PolicyDatum still carries `partner_address = Some(partner_X)` and `partner_share_bps = 1500` for off-chain analytics; on-chain there is no record of which fraction the absorbed partner would have received.

**Crucial delta vs V12.** V12 used `net_premium = premium − raw_fee` (no floor in carve, separate submitter-paid floor pad layered on the team/partner outputs). V12.2 uses `net_pool_growth = premium − fee_total` (floor INSIDE carve, no separate pad). The user-out-of-pocket simplifies from `premium + treasury + floor_pad` to just `premium + treasury` — the pool absorbs the floor's economic cost via slightly-reduced `net_pool_growth` on small premiums, in exchange for a clean linear premium curve that users can predict without the per-premium step function the V12 floor pad introduced.

**Why the change matters economically.** On small premiums V12.2 reduces `net_pool_growth` relative to V12 — for a 20-ADA premium V12 grew the pool by `0.98 × 20 = 19.6 ADA` while V12.2 grows it by 18 ADA. The 1.6-ADA delta is the lovelace V12 previously asked the submitter to subsidize as a separate floor pad; V12.2 has the LP absorb it via reduced net growth in exchange for a flat user-facing premium curve. The effective LP yield on a 20-ADA premium drops from 98% to 90% of premium; on a 100-ADA premium both V12 and V12.2 yield 98% (the floor doesn't activate). The breakeven for "no LP yield delta" is `premium ≥ min_utxo / fee_bps × 10_000 = 2_000_000 / 200 × 10_000 = 100_000_000 lovelace = 100 ADA`. Below 100 ADA, LP gives back a bit to keep the user-facing math clean.

**Why now.** V12 shipped fast (V12.0 → V12.1 → V12.2 in three days under hackathon-grade time pressure). The V12 floor-pad UX created a question every preprod tester asked: "Why does my 20-ADA policy cost 22.5 ADA when the fee is supposed to be 2.5%?" The answer ("0.5% treasury + 1.6 ADA floor pad on the team output") was technically correct but felt arbitrary. V12.2 simplifies the surface area for the VC pitch and the auditor — one premium-driven number, three deterministic outputs, no submitter-state-dependent pads.

**Why also soft-disable Charli3 + Orcfax now.** Both providers are distressed (Charli3 sale signal 2026-04-30, Orcfax wind-down 2026-07-31; see `project_oracle_landscape_2026-05.md`). V12 kept all three providers wired for redundancy, but in practice every preprod policy at the time of writing binds to AegisSelf — neither provider has shipped a working production-grade preprod feed for our path in months. Keeping them enabled in V12.2 is dead surface area and an audit liability ("are we sure the Charli3 path still works correctly?"). Soft-disabling — i.e., the validator rejects new bindings to Charli3/Orcfax but the variants stay in the enum so V11 datums decode — gives clean sunset semantics. We can reactivate either provider in a future redeploy by flipping `is_canonical_oracle_nft` to return `True` for the relevant arm (no schema change, just a flag).

**Why Indigo as a 4th OracleProvider, not AegisSelf-wraps-iUSD.** The original V12.2 plan (spec landed 2026-05-12 morning, commit `ac14ac5`) had AegisSelf publishing iUSD via the Indigo Analytics REST API composed with our existing ADA/USD feed. Wave 1.5 research (this revision) discovered that Indigo already runs their own on-chain oracle — separate from the iAsset CDP registry at `addr1wyyqtkz5rken7jzptp076np606r79lmsrqjrqw8sdn4kvrqewrkdg` (the "iAssetDatum" UTxO the Analytics API points at) — with one parameterized Plutus V2 oracle script per iAsset, each carrying a canonical NFT and a `MkOracleDatum { odPrice, odExpiration }` datum (source: `github.com/IndigoProtocol/indigo-smart-contracts/src/Indigo/Contracts/Oracle/Common.hs`). Live mainnet probe (Koios, Wave 1.5) confirmed the four price-bearing oracle UTxOs are at FOUR DIFFERENT script addresses (one per iAsset, due to the `OracleParams { opOwner, opBiasTime, opExpirationTime }` parameterization), each carrying its canonical NFT and the simple two-field oracle datum. Binding to Indigo's on-chain oracle directly is strictly better than wrapping their REST API: (1) we delete the trust hop on Indigo's HTTPS-server liveness, (2) we expand from 1 iAsset (iUSD only) to 4 (iUSD/iBTC/iETH/iSOL) without adding publisher minting work, (3) the iBTC/iETH/iSOL coverage opens a new product surface (synthetic-asset crash protection), and (4) on-chain binding makes the Aegis validator the only trust hop — same trust model as the existing Charli3/Orcfax/AegisSelf bindings. The off-chain Indigo Analytics REST API (consumed in `api/lending/indigo.py` for CDP browse) remains in use for the LendingProtect "browse at-risk CDPs" flow; it's unrelated to the V12.2 validator binding.

USDA + USDM stay registered but unlit because they're not Indigo iAssets — Indigo doesn't publish an on-chain oracle for them. They remain in the registry with a Coming-soon pill (no specific protocol named — Anzens for USDA and Mehen for USDM are the natural data sources but committing to either is a separate v12.3 work item).

**Why a single coordinated redeploy and not a phased rollout.** Each of the four changes touches at least one of `lib/aegis/types.ak`, `validators/pool.ak`, or `lib/aegis/pricing.ak` — all three are imported by `pool_validator` and rotate its hash on any source change. So any change here is a redeploy regardless. Bundling all four into one deploy (1) keeps the auditor surface area small (one diff to review, one set of hashes to verify, one Cardanoscan walk), (2) avoids the V11→V12 stranded-policy story playing out twice in a week, and (3) lets us correlate the four changes' tests against a single chain state.

## 2. Background

### 2.1 V12 fee flow recap

The V12 Underwrite branch (per `pool.ak:591-734` at V12 HEAD `751c00c`) computes:

```aiken
let net = net_premium(premium, datum.protocol_fee_bps)
// net = premium − floor(premium * fee_bps / 10_000) = premium − raw_fee

let (team_cut, partner_cut) =
  calculate_protocol_fee_split(
    premium,
    datum.protocol_fee_bps,
    new_partner_share_bps,
  )
// team_cut + partner_cut == raw_fee exactly

let team_required_lovelace = floor_to_min_utxo(team_cut)
//   = max(min_utxo_lovelace, team_cut)
let partner_required_lovelace = floor_to_min_utxo(partner_cut)
//   = max(min_utxo_lovelace, partner_cut)

let value_ok =
  assets.lovelace_of(cont_output.value) == assets.lovelace_of(own_value) + net
```

`net_premium` is `premium − raw_fee` (V12's `pricing.ak::net_premium`, line 56). The pool grows by `net` (= 0.98 × premium for the 200-bps default). The team/partner outputs are each floored *separately* via `floor_to_min_utxo` — a min-utxo pad on top of the percentage cut. The submitter's wallet pays the pad (validator-side `>=` check on the output's lovelace; submitter sources the lovelace via wallet inputs, change shows up reduced by the pad amount).

V12's invariant chain for a 20-ADA premium, no partner:

| Lovelace flow | Amount |
|---|---|
| `premium` | 20 ADA (user → consumed) |
| `raw_fee = premium × 0.02` | 0.4 ADA |
| `team_cut = raw_fee − partner_cut` | 0.4 ADA |
| `partner_cut` | 0 |
| `net = premium − raw_fee` | 19.6 ADA → pool continuation |
| `team_required_lovelace = max(2, 0.4)` | 2 ADA → team output |
| `treasury_cut` (Conway donation) | 0.10 ADA |
| User pays = premium + treasury + (team_pad = 2 − 0.4) + tx_fee | 20 + 0.10 + 1.6 + tx_fee |

The 1.6-ADA "team min-utxo floor pad" is the user-facing UX wart — the BuyPanel surfaces it explicitly (see `frontend/src/components/panels/BuyPanel.tsx:431-436`), but every preprod tester asks "why is my 20-ADA policy costing 22.6 ADA?"

### 2.2 V12 floor-pad UX confusion

The current BuyPanel (HEAD `751c00c`) shows:

```
Premium                                20.00 ADA
↳ Net to pool (98%)                    19.60 ADA
↳ Team fee (2%)                         0.40 ADA
Treasury donation (0.5%)               +0.10 ADA
Team min-utxo floor pad                +1.60 ADA   ← shown ONLY when premium × 0.02 < 2
You pay (excl. tx fee)                 22.10 ADA
```

The conditional row (line 431 of `BuyPanel.tsx`) is correct but reads as a hidden surcharge. Every operator demo we've run has had to explain it. V12.2 removes the row by moving the floor *inside* the fee_total computation; the row becomes mathematically inactive.

V12 spec §13.1 notes the user-visible fee jump from V11's 0.5% to V12's 2.5% as "the first version where the team collects revenue" — V12.2 keeps the 2.5% advertised rate but the *effective* user cost on small premiums goes UP slightly (a 20-ADA policy pays a 10.5% effective fee in V12.2 vs ~10.8% in V12 — because the LP gives 1.6 ADA back to subsidize the clean math). The LP yield on small premiums goes DOWN slightly. Net-net: small-premium economics shift LP-ward; large-premium economics are identical.

Trade-off captured in §10.1.

### 2.3 V12 oracle pin

V12's `is_canonical_oracle_nft` (per `oracle.ak:105-114` at HEAD `751c00c`):

```aiken
pub fn is_canonical_oracle_nft(
  provider: OracleProvider,
  oracle_nft: ByteArray,
) -> Bool {
  when provider is {
    Charli3 -> oracle_nft == charli3_ada_usd_nft_policy
    Orcfax -> oracle_nft == orcfax_fsp_script_hash
    AegisSelf -> list.has(aegis_self_canonical_nfts, oracle_nft)
  }
}
```

Charli3 is pinned to a single ADA-only NFT (the `charli3_ada_usd_nft_policy` constant, `types.ak:339`). Orcfax is pinned to a single FSP script hash (`orcfax_fsp_script_hash`, `types.ak:428`). Neither has any path for adding a second pair without a validator-hash rotation. AegisSelf is the only provider with a 5-element allowlist.

In practice every V12 preprod policy binds to AegisSelf (the operator's testing path; the dApp's BuyPanel and DepegPanel build endpoints both route through AegisSelf for ADA + the 5 supported assets). Charli3 and Orcfax are dead code at the runtime call-graph layer.

### 2.6 Indigo on-chain oracle discovery (Wave 1.5 pivot)

This subsection records architectural facts validated 2026-05-12 via Koios mainnet probes. They invalidate the original Wave-1 plan ("AegisSelf wraps iUSD via Indigo Analytics REST API"); the V12.2 plan is now "Indigo as 4th OracleProvider, 4 iAssets directly bound."

**Indigo's on-chain architecture.** Indigo's contracts (source: `github.com/IndigoProtocol/indigo-smart-contracts`, Plutus V2 Haskell, BUSL-1.1 licensed) include a parameterized `Oracle` validator. The relevant pieces:

```haskell
-- src/Indigo/Contracts/Oracle/Common.hs
data OracleParams = OracleParams
  { opOwner          :: PubKeyHash    -- the keeper VKH that signs every refresh
  , opBiasTime       :: POSIXTime     -- validity-range slack
  , opExpirationTime :: POSIXTime     -- how far ahead the keeper may schedule expiry
  }
-- PlutusTx.makeIsDataIndexed ''OracleParams [('OracleParams, 0)]

data OracleDatum = MkOracleDatum
  { odPrice      :: OnChainDecimal   -- newtype Integer; 1e6 scale (see Indigo.Data.Decimal)
  , odExpiration :: POSIXTime        -- POSIX ms
  }
-- PlutusTx.makeIsDataIndexed ''OracleDatum [('MkOracleDatum, 0)]

data OracleRedeemer = FeedPrice POSIXTime OnChainDecimal
-- PlutusTx.makeIsDataIndexed ''OracleRedeemer [('FeedPrice, 0)]
```

The `Oracle` validator is parameterized by `OracleParams`. Each iAsset gets its own instance (different `opOwner` PubKeyHash → different parameterized hash → different mainnet script address). The validator enforces three rules on `FeedPrice now price`:

1. `txSignedBy info opOwner` — the keeper's signature is required on every refresh tx (one of two trust layers).
2. The continuing output must be `MkOracleDatum price (odExpiration outputDatum)` at the same address with the same value (NFT preservation).
3. `price > 0` AND `expiration` falls inside `[currentTimeApprox, currentTimeApprox + opExpirationTime]` AND `now` falls inside the tx's validity range bounded by `opBiasTime`.

**Trust handshake — three layers (vs Aegis's existing four).** Indigo's parser MUST enforce:

1. **NFT pin.** The UTxO must carry a token under one of the 4 canonical Indigo oracle NFT policy ids pinned in `lib/aegis/types.ak::indigo_canonical_nfts`. This is the first trust layer — an attacker forging a fake oracle UTxO under their own permissive policy is rejected before any other check.
2. **Script-address pin.** The UTxO's payment credential must be a `Script(h)` where `h` matches the per-iAsset oracle script hash in `lib/aegis/types.ak::indigo_oracle_script_hashes`. Why pin the script too: a stolen NFT moved to an attacker's wallet UTxO would pass the NFT check; the script-credential pin closes that surface (parallel to Orcfax's "NFT policy == script hash" pattern and AegisSelf's "publisher VKH credential" pin).
3. **Quantity = 1.** Each NFT is a one-shot mint (Indigo's deployment). The parser asserts `quantity_of(value, policy, asset_name) == 1` for the configured (policy, asset_name) pair.

We do NOT pin Indigo's keeper VKH (`opOwner`). The reason: Indigo's keeper-VKH-per-iAsset is opaque to consumers (not published in their docs as a stable contract); the keeper key may rotate over time; and Indigo's own validator already enforces `txSignedBy info opOwner` at refresh time — so any UTxO that lands at the canonical script address with the canonical NFT IS by construction a keeper-signed refresh (or it wouldn't have validated). Pinning the script hash + the NFT IS the keeper handshake, transitively.

**Freshness cadence (empirical).** Live mainnet probe of the iUSD oracle's tx history at script credential `1042576d1de963437bd4bb50dcd91459756e6812b7c06dd543a332c8` (1000 most recent txs via Koios `credential_txs`):

- Mean inter-refresh delta: **2130 s ≈ 35.5 min**
- Median: ~40 min
- Min: 315 s (rare burst), max: 7931 s (occasional gap)
- The `odExpiration` in the datum is set ~49 min ahead of the publish block time (= `opExpirationTime` configured value).

Aegis's existing consumer-side freshness window for Charli3/Orcfax/AegisSelf is 65–70 min. Indigo's effective publish→expiry window is 49 min. Since the parser reads `odExpiration` directly and the validator checks `tx_lower <= datum.expiry`, **we do NOT impose an Aegis-side freshness window** for Indigo — we trust Indigo's own expiration field. This is simpler than Orcfax (which has no on-chain expiry, so Aegis derives one) and matches AegisSelf (which writes its own expiry field). The 49-min window comfortably covers preprod tx propagation latency (typically < 1 min). If Indigo's keepers stall, claims will reject at the freshness gate, but no operator action is required.

**Mainnet vs preprod.** Probed `preprod.analytics.indigoprotocol.io`, `preview.analytics.indigoprotocol.io`, `staging.analytics.indigoprotocol.io` — all three DNS-fail. Indigo is mainnet-only. We deploy a preprod mock in Wave 4 (§5b) so the dev loop, soak, and Playwright e2e can exercise Indigo-bound paths without touching mainnet.

**Network-specific NFT policy ids — why we can't reuse mainnet ids on preprod.** Indigo's NFT policies are minted from a one-shot policy parameterized by a consumed init UTxO (CIP-67 reference NFT pattern; the asset name in their `outputNftTn` field IS the CIP-67 reference label hex-encoded with a datestamp). Reproducing the same 28-byte policy id on preprod would require: (1) consuming the same init UTxO on preprod (impossible — UTxOs are network-local), OR (2) deploying Indigo's full Plutus toolchain on preprod (Indigo doesn't run preprod). So the preprod mock NFTs use FRESH one-shot policy ids minted by `scripts/init_indigo_mock.py` (Wave 4) under the operator's mock script. The validator's `indigo_canonical_nfts` constant is therefore per-network (preprod entries = 4 mock-mint hashes, mainnet entries = the 4 production hashes in the table above) — same pattern as `charli3_ada_usd_nft_policy` which is per-network today.

**Decision: validator is denomination-agnostic.** Indigo prices are ADA-denominated (per `Indigo.Data.Decimal.decimalToAdaValue`). The V12.2 validator does NOT introduce an ADA/USD cross-rate for Indigo bindings — the validator simply compares `Indigo.resolve(...).value` (ADA per iAsset in 1e6 scale) to the strike (caller-provided, same scale). The off-chain UI is responsible for converting strike between USD and ADA-denominated units for display. This matches the existing V12 convention: the validator never knows what unit a strike is in; it just compares numbers. The off-chain `policies.py` and frontend `BuyPanel.tsx` enforce that the strike denomination matches the bound oracle's denomination by-construction (the oracle resolver dictates the unit). This is the cleanest possible separation — formalised in §3.4a invariant `INDIGO-INV-1`.

### 3.4 `lib/aegis/pricing.ak` — Hybrid fee helper rewrite

The current V12 `pricing.ak` exposes:

- `calculate_protocol_fee(premium, fee_bps) -> Int` (line 51) — `premium × fee_bps / 10_000`, i.e., `raw_fee`.
- `net_premium(premium, fee_bps) -> Int` (line 56) — `premium − raw_fee`.
- `calculate_protocol_fee_split(premium, fee_bps, partner_share_bps) -> (Int, Int)` (line 102) — splits `raw_fee` into `(team_cut, partner_cut)` with `team + partner == raw_fee` exactly.
- `calculate_treasury_cut(premium, fee_bps, treasury_share_bps) -> Int` (line 76) — `premium × fee_bps × treasury_share_bps / 1e8`, unchanged in V12.2.

V12.2 keeps `calculate_protocol_fee` (raw rate), keeps `calculate_treasury_cut` (treasury math unchanged), and rewrites the other two:

- **New** `calculate_fee_total(premium, fee_bps, min_utxo_lovelace) -> Int` — returns `max(min_utxo_lovelace, calculate_protocol_fee(premium, fee_bps))`. This is the "floor INSIDE the carve" value.
- **New** `calculate_net_pool_growth(premium, fee_bps, min_utxo_lovelace) -> Int` — returns `premium − calculate_fee_total(...)`. The validator's pool-value invariant reads from this.
- **Rewrite** `calculate_protocol_fee_split` — now takes `fee_total` as input (rather than recomputing `raw_fee`), and applies the **silent partner_cut absorb** rule: sub-floor partner_cut → 0, team_cut absorbs the leftover.
- **Deprecate** `net_premium` — keep the function in tree for the test block and for cross-validator analytics, but mark with a comment "// [v12.2] DEPRECATED — V12.2 validators use calculate_net_pool_growth instead. V11/V12 callers continue to read this for legacy analytics." The function still compiles and tests still pass.

**Diff against `lib/aegis/pricing.ak`**:

```aiken
// ---------------------------------------------------------------------------
// Protocol Fee Calculation (V12.2: Hybrid Fee — floor inside carve)
// ---------------------------------------------------------------------------

/// Calculate the raw (un-floored) protocol fee from a premium.
/// Fee is denominated in basis points (1 bp = 0.01%).
/// Example: 200 bps on 10 ADA premium = 0.2 ADA raw_fee.
///
/// V12.2 callers usually want `calculate_fee_total` instead (which applies
/// the min-utxo floor). This helper is the un-floored value, used for:
///   * The treasury-cut math (which is treasury_share_bps of the RAW fee,
///     not the floored fee).
///   * Off-chain analytics that want the "intended" fee separate from the
///     floor-padded value.
pub fn calculate_protocol_fee(premium: Int, fee_bps: Int) -> Int {
  premium * fee_bps / 10_000
}

/// [v12.2] Compute the floored fee_total: `max(min_utxo_lovelace, raw_fee)`.
///
/// V12.2's "Hybrid Fee" rule moves the 2-ADA min-utxo floor INSIDE the carve,
/// so the pool's value reduction equals exactly `fee_total` (not just the
/// percentage fee with a separate exogenous pad as in V12). The user's outflow
/// equals `premium + treasury_cut + tx_fee` (linear in premium), with no
/// premium-dependent step function from the V12 floor pad layer.
///
/// Trade-off vs V12: small premiums (below `min_utxo × 10_000 / fee_bps` =
/// 100 ADA at the default 200 bps) yield slightly less net_pool_growth — the
/// LP absorbs the floor cost in exchange for a clean user-facing fee curve.
/// Premiums at or above 100 ADA see `fee_total == raw_fee` and the math is
/// identical to V12. See §10.1 for the full economic analysis.
pub fn calculate_fee_total(
  premium: Int,
  fee_bps: Int,
  min_utxo_lovelace: Int,
) -> Int {
  let raw_fee = calculate_protocol_fee(premium, fee_bps)
  if raw_fee > min_utxo_lovelace {
    raw_fee
  } else {
    min_utxo_lovelace
  }
}

/// [v12.2] Compute the net pool growth: `premium − calculate_fee_total(...)`.
///
/// This is the lovelace delta the pool's continuation UTxO grows by on an
/// Underwrite tx (and the negative of what it decreases by on the cancel-path
/// extraction). The validator's `value_ok` check enforces this exactly:
///
///   cont_pool.lovelace == own.lovelace + calculate_net_pool_growth(...)
///
/// On Underwrite, `total_liquidity` in the new pool datum grows by the same
/// amount — pool accounting stays honest (no phantom liquidity, same as V12).
pub fn calculate_net_pool_growth(
  premium: Int,
  fee_bps: Int,
  min_utxo_lovelace: Int,
) -> Int {
  premium - calculate_fee_total(premium, fee_bps, min_utxo_lovelace)
}

/// [v12.2] DEPRECATED — V12.2 validators use `calculate_net_pool_growth`
/// instead. V11/V12 callers continue to read this for legacy analytics; the
/// function stays compiled so cross-version code paths keep working.
///
/// V11 / V12 semantics: `premium − raw_fee` (no floor in the carve).
/// V12.2 semantics: callers should switch to `calculate_net_pool_growth`
/// which applies the floor inside the carve. The two functions agree for
/// `premium >= min_utxo × 10_000 / fee_bps`.
pub fn net_premium(premium: Int, fee_bps: Int) -> Int {
  premium - calculate_protocol_fee(premium, fee_bps)
}

/// [v12.2] Split a `fee_total` into team and partner cuts with silent
/// partner_cut absorb.
///
/// Returns `(team_cut, partner_cut)` such that:
///   `team_cut + partner_cut == fee_total` (always, exactly)
/// and
///   `partner_cut == 0 OR partner_cut >= min_utxo_lovelace` (silent absorb).
///
/// The silent absorb rule: when `partner_cut_raw = fee_total × partner_share_bps
/// / 10_000 < min_utxo_lovelace`, the partner output is NOT emitted on chain
/// and `team_cut` absorbs the full `fee_total`. PolicyDatum still records
/// `partner_address` and `partner_share_bps` for off-chain analytics — the
/// partner registry can reconstruct "would have received" amounts from the
/// datum + premium for any cancelled / fully-attributed analysis.
///
/// Why silent absorb instead of submitter-paid floor pad (V12 behavior):
/// the floor pad UX was confusing (every preprod tester asked "why does my
/// 20-ADA policy cost 22.5 ADA?"). Absorb keeps the user-facing math linear
/// and pushes the cost back into the team_cut budget (which absorbs without
/// any user-visible effect). team_address is compile-time pinned (V12 §3.5),
/// so the team_cut value never diverts away from the pinned address — no
/// attack surface (§10.3).
///
/// Floor-rounding convention: `partner_cut_raw` is integer-divided (floored)
/// per the standard Aiken `/` operator. The `partner_cut < min_utxo` test
/// uses the SAME `min_utxo_lovelace` value the validator passes in, so the
/// off-chain `add_protocol_fee_outputs` helper and the on-chain check produce
/// identical (team_cut, partner_cut) tuples bit-for-bit.
pub fn calculate_protocol_fee_split(
  fee_total: Int,
  partner_share_bps: Int,
  min_utxo_lovelace: Int,
) -> (Int, Int) {
  let partner_cut_raw = fee_total * partner_share_bps / 10_000
  let partner_cut =
    if partner_cut_raw >= min_utxo_lovelace {
      partner_cut_raw
    } else {
      0
    }
  let team_cut = fee_total - partner_cut
  (team_cut, partner_cut)
}
```

**Signature change.** V12's `calculate_protocol_fee_split` took `(premium, protocol_fee_bps, partner_share_bps)`; V12.2's takes `(fee_total, partner_share_bps, min_utxo_lovelace)`. This is a deliberate breaking change at the helper-signature layer — V12.2 wants the floored `fee_total` to be computed once at the call site (so the validator and off-chain math share the same input to the split), and the silent absorb rule needs `min_utxo_lovelace` as input. The Wave 2 agent updates the three validator branches (§3.5–3.7) to pass the new signature.

### 3.4a `lib/aegis/oracle/indigo.ak` (NEW) — Indigo on-chain oracle parser

The Indigo parser mirrors `oracle/aegis_self.ak`'s find-feed-output + find-feed-datum + resolve pattern. Two key differences vs AegisSelf:

1. **Per-iAsset script credential pin** (not a single shared publisher VKH). The script hash is looked up from `indigo_oracle_script_hashes` using the index of the matching `oracle_nft` in `indigo_canonical_nfts`. (Why: each iAsset's Indigo Oracle validator is parameterized differently — see §2.6.)
2. **Datum shape: `Constr_0[Constr_0[price_int], expiration_ms]`** (Plutus V2 `OracleDatum`), NOT Charli3's `Tag 121 → Tag 123 → {0: price, 1: created_ms, 2: expiry_ms}`. The Indigo datum is two fields (price + expiration) vs Charli3's three (price + created + expiry); the wrapping Constr_0 around `OnChainDecimal` adds one nesting layer.

**File path:** `contracts/lib/aegis/oracle/indigo.ak`

**Full Aiken source (Wave 2 implements):**

```aiken
// =============================================================================
// Aegis Insurance Protocol - Indigo Oracle Parser (v12.2 fourth provider)
// Location: contracts/lib/aegis/oracle/indigo.ak
// Summary: Indigo on-chain oracle resolver. Reads a UTxO at the per-iAsset
//          parameterized Indigo Oracle script address carrying the canonical
//          Oracle Asset NFT and parses its inline datum as the Plutus V2
//          `OracleDatum = Constr_0[OnChainDecimal, POSIXTime]` shape.
//
//          Trust handshake (three layers):
//            1. UTxO must carry a token under the supplied `oracle_nft` policy
//               id (= one of the 4 canonical Indigo NFTs per network).
//            2. UTxO's payment credential must be `Script(h)` where h equals
//               the per-iAsset oracle script hash paired with the NFT in
//               `indigo_oracle_script_hashes` by list index.
//            3. Quantity of the (policy, asset_name) NFT in the UTxO must
//               equal 1 (Indigo's one-shot mint guarantee).
//
//          Why three layers: layer 1 prevents fake NFTs minted under a
//          permissive attacker policy. Layer 2 prevents a stolen NFT lying
//          at a non-canonical address. Layer 3 defends against quantity-0
//          phantom matches (defense in depth; Indigo's validator already
//          enforces continuing-output preservation of value, but Aegis
//          re-checks since we are the consumer).
//
//          Why we do NOT pin Indigo's keeper VKH: the keeper-VKH per iAsset
//          is opaque to consumers (Indigo treats it as their internal
//          rotation key). Indigo's own `validateOracle` already enforces
//          `txSignedBy info opOwner` on every refresh, so any UTxO at the
//          canonical script address with the canonical NFT is by construction
//          a keeper-signed refresh. The script-credential pin IS the
//          keeper-handshake transitively.
//
// References:
//   * Indigo Oracle source: github.com/IndigoProtocol/indigo-smart-contracts
//                           /src/Indigo/Contracts/Oracle/Common.hs
//   * Indigo Decimal:       /src/Indigo/Data/Decimal.hs (decimalUnit = 1e6)
//   * V12.2 spec §2.6, §3.4a, §10.X
// =============================================================================

use aegis/oracle/types.{Price}
use aegis/types as aegis_types
use aiken/collection/list
use cardano/address.{Script}
use cardano/assets
use cardano/transaction.{InlineDatum, Input, Output}

// ---------------------------------------------------------------------------
// Indigo Datum Shape (re-declared positionally to match upstream CBOR)
// ---------------------------------------------------------------------------

/// Indigo's `OnChainDecimal` is a newtype Integer wrapped in Constr_0. The
/// scale is 1e6 (see `Indigo.Data.Decimal.decimalUnit`). We re-declare
/// positionally so this parser does not depend on an external Indigo Aiken
/// library (Indigo's contracts are Plutus V2 Haskell — no Aiken bindings
/// shipped by upstream). The CBOR wire format is `d8 79 9f <int> ff`.
pub type OnChainDecimal {
  value: Int,
}

/// Indigo's `OracleDatum` matching `MkOracleDatum :: OnChainDecimal -> POSIXTime`.
/// CBOR wire format: `d8 79 9f <Constr_0[int]> <int> ff` —
/// Constr_0[OnChainDecimal, expiration_ms].
///
/// The price field is ADA-denominated (Indigo's `decimalToAdaValue`); the
/// expiration is POSIX milliseconds. No `created_at` field — Indigo writes
/// only the future expiration, not the publish timestamp.
pub type IndigoOracleDatum {
  price: OnChainDecimal,
  expiration: Int,
}

// Canonical-feed UTxO finder
// ---------------------------------------------------------------------------

/// Look up the per-iAsset oracle script hash paired with `oracle_nft` in
/// `indigo_canonical_nfts`. The two lists are paired by list index; this
/// helper walks both in lockstep to find the matching entry. Returns
/// `Some(script_hash)` on hit, `None` if `oracle_nft` is not in the
/// canonical NFT list.
///
/// Aiken-time complexity: O(n) over the canonical list, n=4 in V12.2.
fn resolve_paired_script_hash(
  oracle_nft: ByteArray,
) -> Option<ByteArray> {
  // Pseudocode walking both lists by index. Wave 2 may rewrite as a single
  // `list.zip + list.find` for clarity.
  do_resolve_paired(
    aegis_types.indigo_canonical_nfts,
    aegis_types.indigo_oracle_script_hashes,
    oracle_nft,
  )
}

fn do_resolve_paired(
  nfts: List<ByteArray>,
  hashes: List<ByteArray>,
  target: ByteArray,
) -> Option<ByteArray> {
  when nfts is {
    [] -> None
    [nft, ..rest_nfts] ->
      when hashes is {
        [] -> None  // Defensive: should never happen if lists same length.
        [hash, ..rest_hashes] ->
          if nft == target {
            Some(hash)
          } else {
            do_resolve_paired(rest_nfts, rest_hashes, target)
          }
      }
  }
}

/// Search `reference_inputs` for the canonical Indigo oracle UTxO for the
/// given `oracle_nft`. A UTxO qualifies iff it (a) carries a token under
/// `oracle_nft` policy id with `quantity == 1`, AND (b) sits at a payment
/// credential equal to `Script(h)` where h is the per-iAsset script hash
/// paired with the NFT.
fn find_feed_output(
  reference_inputs: List<Input>,
  oracle_nft: ByteArray,
  expected_script_hash: ByteArray,
) -> Option<Output> {
  when reference_inputs is {
    [] -> None
    [input, ..rest] -> {
      let has_indigo_nft =
        assets.policies(input.output.value)
          |> list.has(oracle_nft)
      let at_indigo_script =
        when input.output.address.payment_credential is {
          Script(h) -> h == expected_script_hash
          _ -> False
        }
      // Note: we check quantity-of inside the `find_feed_datum` for clarity;
      // here we just gate on policy presence + script credential.
      if has_indigo_nft && at_indigo_script {
        Some(input.output)
      } else {
        find_feed_output(rest, oracle_nft, expected_script_hash)
      }
    }
  }
}

/// Locate the canonical Indigo oracle UTxO for `oracle_nft` and decode its
/// inline datum as `IndigoOracleDatum`.
pub fn find_feed_datum(
  reference_inputs: List<Input>,
  oracle_nft: ByteArray,
) -> IndigoOracleDatum {
  // 1. Canonical-NFT pin (layer 1 of the trust handshake).
  expect list.has(aegis_types.indigo_canonical_nfts, oracle_nft)

  // 2. Resolve the paired script hash from the canonical list (layer 2).
  expect Some(expected_script_hash) = resolve_paired_script_hash(oracle_nft)

  // 3. Find the matching UTxO in reference_inputs.
  when find_feed_output(reference_inputs, oracle_nft, expected_script_hash) is {
    Some(output) ->
      when output.datum is {
        InlineDatum(raw) -> {
          // 4. Quantity-1 check (layer 3 of the trust handshake). We do this
          // after the find because find_feed_output already gated on policy
          // presence; here we just check the quantity is exactly 1 (not
          // possible to be 0 since policy is present, but 2+ would be an
          // attacker minting extra tokens under a permissive policy — which
          // is impossible for Indigo's one-shot policies, but defense in
          // depth costs nothing).
          //
          // The asset_name is read from a (policy_id, asset_name) entry; in
          // practice Indigo's `outputNftTn` is per-iAsset (e.g., the iUSD
          // NFT's asset name is "iUSD20221118195906"). Indigo's policy ids
          // are also per-iAsset (different policies per iAsset on mainnet)
          // so the (policy, *) wildcard sum across asset names equals 1 in
          // every observed UTxO. The parser checks `quantity_of(policy, *) == 1`
          // by computing `tokens(value, policy)` and asserting list length 1
          // with quantity 1.
          expect single_token_of_policy(output.value, oracle_nft)
          expect datum: IndigoOracleDatum = raw
          datum
        }
        _ -> fail @"Indigo feed UTxO does not have inline datum"
      }
    None -> fail @"Indigo feed UTxO not found in reference inputs"
  }
}

/// Assert that the value carries EXACTLY one token under `policy_id` (across
/// all asset names). Indigo's one-shot policies guarantee this; we re-check
/// at the consumer side for defense in depth.
///
/// Returns Bool so the caller can `expect` on it. Wave 2 implements via
/// `assets.tokens(value, policy_id)` returning a Dict, then folding
/// quantities to a total. The Aiken implementation:
///
/// ```aiken
/// fn single_token_of_policy(value: Value, policy_id: ByteArray) -> Bool {
///   let tokens_under_policy = assets.tokens(value, policy_id)
///   let total =
///     dict.foldr(tokens_under_policy, 0, fn(_name, qty, acc) { acc + qty })
///   total == 1
/// }
/// ```
///
/// At the call site:
///
/// ```aiken
/// expect single_token_of_policy(output.value, oracle_nft)
/// ```

// ---------------------------------------------------------------------------
// Provider-uniform Resolver (v12.2 entrypoint)
// ---------------------------------------------------------------------------

/// Resolve an Indigo oracle reading from `reference_inputs` and emit the
/// provider-uniform `Price` record.
///
/// **Trust handshake.** See `find_feed_datum` — NFT presence, paired script
/// credential, AND single-token-of-policy check are all enforced.
///
/// **Scaling.** Indigo publishes at 1e6 scale (Plutus V2 OnChainDecimal,
/// `decimalUnit = 1_000_000` per Indigo.Data.Decimal). Same scale as
/// Charli3/AegisSelf/Orcfax post-conversion. No rescaling needed.
///
/// **Denomination.** Indigo prices are ADA-denominated (USD-per-iAsset would
/// require a separate ADA/USD cross-rate; Indigo doesn't publish that). The
/// V12.2 validator is denomination-agnostic — it compares the price value
/// directly to the strike, both in 1e6 scale. The off-chain layer is
/// responsible for asserting that strike denomination matches the oracle's
/// denomination. See §2.6 "Decision: validator is denomination-agnostic."
///
/// **Freshness.** Indigo writes `odExpiration` directly into the datum; we
/// surface that as `Price.valid_until`. No Aegis-side freshness window —
/// the validator's downstream `tx_lower <= valid_until` check is the only
/// freshness gate. Indigo's empirical refresh cadence is ~35 min mean
/// (validated 2026-05-12 against 1000 most recent iUSD oracle txs), with
/// `odExpiration` set ~49 min ahead of publish slot.
///
/// **observed_at.** Indigo's datum has NO publish-timestamp field (unlike
/// Charli3 which writes `created_at` and AegisSelf which inherits that). We
/// derive `observed_at` by subtracting Indigo's typical 49-min lookahead
/// from `expiration`. This is approximate (the lookahead is configurable
/// per-iAsset via `opExpirationTime`) but the consumer doesn't actually
/// USE observed_at for anything load-bearing — only freshness gating cares,
/// and that uses `valid_until` directly. We could also set `observed_at = 0`;
/// the choice doesn't affect any downstream check.
pub fn resolve(reference_inputs: List<Input>, oracle_nft: ByteArray) -> Price {
  let datum = find_feed_datum(reference_inputs, oracle_nft)
  // The Indigo lookahead constant — derived from observed cadence on mainnet.
  // Wave 2 may pin this to a per-network constant in types.ak if needed.
  let indigo_typical_lookahead_ms = 49 * 60 * 1_000
  Price {
    value: datum.price.value,
    observed_at: datum.expiration - indigo_typical_lookahead_ms,
    valid_until: datum.expiration,
  }
}

### 3.5 `validators/pool.ak` Underwrite branch — Hybrid fee invariants

The V12 Underwrite branch (lines 591-734 at HEAD `751c00c`) computes:

```aiken
let net = net_premium(premium, datum.protocol_fee_bps)
let (team_cut, partner_cut) =
  calculate_protocol_fee_split(premium, datum.protocol_fee_bps, new_partner_share_bps)
let team_required_lovelace = floor_to_min_utxo(team_cut)
let partner_required_lovelace = floor_to_min_utxo(partner_cut)
let value_ok = cont.lovelace == own.lovelace + net
```

V12.2 replaces this with:

```aiken
let fee_total =
  calculate_fee_total(premium, datum.protocol_fee_bps, min_utxo_lovelace)
let net_pool_growth = premium - fee_total
let (team_cut, partner_cut) =
  calculate_protocol_fee_split(fee_total, new_partner_share_bps, min_utxo_lovelace)
let team_output_funded = output_to_address_at_least(outputs, team_address, team_cut)
let partner_output_funded =
  when new_partner_address is {
    Some(addr) ->
      if partner_cut > 0 {
        output_to_address_at_least(outputs, addr, partner_cut)
      } else {
        // Silent absorb: partner_cut absorbed into team_cut, no partner output
        // is required even though partner_address is Some(_).
        True
      }
    None -> True
  }
let value_ok = cont.lovelace == own.lovelace + net_pool_growth
```

Note the elimination of `floor_to_min_utxo` calls on the output thresholds — `fee_total` already incorporates the floor inside the carve, and `calculate_protocol_fee_split` already applies the silent absorb rule. Both outputs check `>=` against the unfloored split values; for the team output `team_cut >= fee_total >= min_utxo_lovelace` always (invariant §8.1.4 in the test invariants), so the validator can omit the floor wrapper on team. For the partner output, the silent absorb guarantees `partner_cut == 0 OR partner_cut >= min_utxo_lovelace` (invariant §8.1.3), so the conditional check is exact.

**Full V12.2 Underwrite branch diff** (replaces lines 591-734):

```aiken
// -----------------------------------------------------------------------
// [v12.2 §3.5] UNDERWRITE: Reserve coverage for a new policy, collect
// premium, and route the protocol fee to team_address (+ optional
// partner_address per the new policy's PolicyDatum).
//
// V12.2 breaking changes vs V12:
//   * Pool continuation grows by `net_pool_growth = premium − fee_total`
//     where fee_total = max(min_utxo, raw_fee). V12 grew it by
//     `net_premium = premium − raw_fee` (no floor in carve).
//   * Team output threshold is `team_cut` exactly (no separate floor pad).
//     Validator-side invariant guarantees team_cut >= min_utxo_lovelace
//     because team_cut ∈ [fee_total − partner_cut .. fee_total] and
//     fee_total >= min_utxo.
//   * Partner output threshold is `partner_cut` when nonzero; partner
//     output is OMITTED entirely when partner_cut is sub-floor (silent
//     absorb, §3.4 / §10.3).
//   * No floor_to_min_utxo wrapper on either output. The floor is inside
//     fee_total.
// -----------------------------------------------------------------------
Underwrite { coverage, premium } -> {
  expect Some(cont_output) = pool_output

  // 1. Parse the new pool datum
  expect InlineDatum(raw_new_datum) = cont_output.datum
  expect new_datum: PoolDatum = raw_new_datum

  // [FIX A-024] Reject non-positive coverage / premium. Unchanged from V12.
  let coverage_positive = coverage > 0
  let premium_positive = premium > 0

  // 2. Verify premium adequacy
  let premium_ok = is_premium_adequate(premium, coverage)

  // 3. Verify pool can underwrite this coverage
  let can_cover =
    can_underwrite(datum.total_liquidity, datum.active_coverage, coverage)

  // 4. [v12.2] Compute fee_total with floor INSIDE the carve.
  let fee_total =
    calculate_fee_total(premium, datum.protocol_fee_bps, min_utxo_lovelace)
  let net_pool_growth = premium - fee_total

  // 5. Verify datum update is correct. The new datum's total_liquidity
  // grows by net_pool_growth (NOT premium, NOT V12's net_premium).
  let datum_ok =
    verify_underwrite_datum(
      datum.total_liquidity,
      datum.active_coverage,
      new_datum.total_liquidity,
      new_datum.active_coverage,
      net_pool_growth,
      coverage,
    )

  // 6. Verify immutable fields are preserved (incl. lp_supply, which
  // does NOT change on Underwrite — only on Add/Remove Liquidity).
  let immutable_ok =
    new_datum.lp_token_policy == datum.lp_token_policy && new_datum.protocol_fee_bps == datum.protocol_fee_bps && new_datum.pool_nft == datum.pool_nft && new_datum.lp_supply == datum.lp_supply

  // 7. [v12.2 §3.5] Pool value grows by `net_pool_growth`, NOT the full
  // premium and NOT V12's `net_premium`. The fee_total leaves the pool to
  // team/partner outputs; the floor is inside fee_total so the pool
  // gives back a slight discount on small premiums (LP absorbs the floor
  // cost in exchange for a clean user-facing fee curve, §10.1).
  let value_ok =
    assets.lovelace_of(cont_output.value) == assets.lovelace_of(own_value) + net_pool_growth

  // 8. [FIX A-004 / v12] An Underwrite tx MUST create exactly one
  // policy output bound to THIS pool with matching coverage/premium.
  // Helper return type unchanged from V12 (Option<(Option<Address>, Int)>).
  let own_pool_hash =
    when self_input.output.address.payment_credential is {
      Script(h) -> h
      _ -> #""
    }
  expect Some((new_partner_address, new_partner_share_bps)) =
    policy_output_matches_underwrite(
      outputs,
      own_pool_hash,
      datum.pool_nft,
      policy_script_hash,
      validity_range,
      coverage,
      premium,
    )

  // 9. Cap check on partner_share_bps. Unchanged from V12.
  let partner_share_in_bounds =
    new_partner_share_bps >= 0 && new_partner_share_bps <= partner_share_cap_bps

  // 10. Partner-address consistency: nonzero share REQUIRES Some(addr).
  // Unchanged from V12.
  let partner_consistency =
    when new_partner_address is {
      Some(_) -> True
      None -> new_partner_share_bps == 0
    }

  // 11. [v12.2 §3.4] Compute the (team_cut, partner_cut) split with
  // silent partner_cut absorb. team_cut absorbs sub-floor partner_cut.
  let (team_cut, partner_cut) =
    calculate_protocol_fee_split(
      fee_total,
      new_partner_share_bps,
      min_utxo_lovelace,
    )

  // 12. [v12.2 §3.5] Team output check. Threshold is `team_cut` exactly —
  // no floor_to_min_utxo wrapper. team_cut >= min_utxo_lovelace is
  // guaranteed by the silent absorb invariant (§8.1.4).
  let team_output_funded =
    output_to_address_at_least(outputs, team_address, team_cut)

  // 13. [v12.2 §3.5 + §3.4] Partner output check with silent absorb. When
  // partner_cut == 0 (silently absorbed), the partner output is OMITTED
  // entirely — even if partner_address is Some(_), no on-chain output to
  // partner_address is required. PolicyDatum still records the partner
  // attribution for off-chain analytics.
  let partner_output_funded =
    when new_partner_address is {
      Some(addr) ->
        if partner_cut > 0 {
          output_to_address_at_least(outputs, addr, partner_cut)
        } else {
          // Silent absorb path: no partner output required.
          True
        }
      None -> True
    }

  // 14. [FIX A-021] Treasury donation cut. Numerically unchanged from V12.
  // Computed against the RAW premium (not fee_total), so the user sees
  // 0.5% of premium regardless of whether the fee floor activates.
  let required_donation =
    calculate_treasury_cut(
      premium,
      datum.protocol_fee_bps,
      treasury_share_bps,
    )
  let donation_ok =
    when treasury_donation is {
      Some(amt) -> amt >= required_donation
      None -> required_donation == 0
    }

  coverage_positive && premium_positive && premium_ok && can_cover && datum_ok && immutable_ok && value_ok && donation_ok && partner_share_in_bounds && partner_consistency && team_output_funded && partner_output_funded
}

### 3.6 `validators/pool.ak` AcceptCancellation branch — Hybrid fee on cancellation_fee retention

The V12 AcceptCancellation branch (lines 1158-1298) applies the protocol fee split to the cancellation_fee retention (= 10% of premium) — see V12 spec §3.9 (B2 design). V12.2 keeps that design intent but applies the Hybrid fee math to the cancellation_fee:

```aiken
let cancellation_fee = policy_datum.premium_paid - refund_amount

// V12 computed:
//   let (team_cut, partner_cut) = calculate_protocol_fee_split(
//     cancellation_fee, datum.protocol_fee_bps, policy_datum.partner_share_bps)
//   let team_required_lovelace = floor_to_min_utxo(team_cut)
//   let partner_required_lovelace = floor_to_min_utxo(partner_cut)
//   value_ok: cont == own - refund_amount - team_cut - partner_cut

// V12.2 computes:
let fee_total_cancel =
  calculate_fee_total(
    cancellation_fee, datum.protocol_fee_bps, min_utxo_lovelace)
let (team_cut, partner_cut) =
  calculate_protocol_fee_split(
    fee_total_cancel, policy_datum.partner_share_bps, min_utxo_lovelace)
// team_output threshold = team_cut (no wrapper); partner conditionally on partner_cut > 0
// value_ok: cont == own - refund_amount - fee_total_cancel
```

The cancel-path fee_total is computed against the *cancellation_fee* (10% of premium), not the original premium. So for a 100-ADA cancelled policy:
- `cancellation_fee = 10 ADA`
- `raw_fee_cancel = 10 × 200 / 10_000 = 0.2 ADA`
- `fee_total_cancel = max(2, 0.2) = 2 ADA`
- `net_pool_loss = refund_amount + fee_total_cancel = 90 + 2 = 92 ADA`

The pool ends up keeping `100 − 92 = 8 ADA` from a cancelled 100-ADA policy. V12 kept `100 − 90 − 0.2 − 0 = 9.8 ADA` (LP kept more because of no floor-in-carve). V12.2's design has the LP give back 1.8 ADA more on small-premium cancels in exchange for the clean user-facing math. For 500-ADA premium cancels, `cancellation_fee = 50 ADA`, `fee_total_cancel = 1 ADA` (raw) — wait, no: `0.02 × 50 = 1 ADA` which is BELOW the 2-ADA floor, so `fee_total_cancel = 2 ADA`. LP keeps `500 − 450 − 2 = 48 ADA`. For 1000-ADA: cancellation_fee = 100 ADA, raw_fee_cancel = 2 ADA (exactly at floor), `fee_total_cancel = 2`, LP keeps `1000 − 900 − 2 = 98 ADA`. The cancel-path floor takes a bite out of LP cancel yield specifically because `cancellation_fee = 0.1 × premium` keeps raw_fee_cancel low even on premiums above the Underwrite-time floor breakeven.

### 3.7 `validators/pool.ak` BatchUnderwrite branch — Hybrid fee aggregation

The V12 BatchUnderwrite branch (lines 939-1068) walks the per-policy outputs via `batch_policies_match_totals_v12` and aggregates floored team and partner cuts. V12.2 simplifies the aggregation by adopting the Hybrid fee math per-policy:

- Each policy's `fee_total_i = max(min_utxo, premium_i × fee_bps / 10_000)`.
- Each policy's `(team_cut_i, partner_cut_i)` is computed via the V12.2 `calculate_protocol_fee_split` (which applies silent absorb per-policy).
- `team_total = Σ team_cut_i` — no separate floored/unfloored variant. team_cut is already the right value.
- `partner_totals[addr] = Σ_{i: partner_i == addr} partner_cut_i` — entries with `partner_cut_i == 0` (silent absorb) contribute nothing.
- Pool value invariant: `cont_pool.lovelace == own.lovelace + (total_premium − Σ fee_total_i)`.

V12's `BatchFeeTotals` record had four fields tracking floored vs unfloored team/partner sums:

```aiken
type BatchFeeTotals {
  cov_sum: Int,
  prem_sum: Int,
  funded_ok: Bool,
  team_total_floored: Int,      // sum of max(min_utxo, team_cut_i)
  team_total_unfloored: Int,    // sum of team_cut_i (percentage-calculated)
  partner_totals_floored: List<(Address, Int)>,
  partner_totals_unfloored_sum: Int,
  shares_ok: Bool,
}
```

V12.2 collapses to a simpler record:

```aiken
type BatchFeeTotals {
  cov_sum: Int,
  prem_sum: Int,
  funded_ok: Bool,
  fee_total_sum: Int,            // Σ fee_total_i (sum of per-policy floored fee_totals)
  team_total: Int,               // Σ team_cut_i (always == fee_total_sum − partner_totals_sum)
  partner_totals: List<(Address, Int)>,  // Σ partner_cut_i per unique partner address
  partner_totals_sum: Int,       // Σ over all entries of partner_totals (cached for value_ok)
  shares_ok: Bool,
}
```

The collapse drops two fields (`team_total_floored` vs `team_total_unfloored`; `partner_totals_floored` vs `partner_totals_unfloored_sum`) because V12.2's `calculate_protocol_fee_split` already encodes the silent absorb rule — `team_total` and `partner_totals` are the canonical "actual lovelace routed" values, no separate "intended" tracking needed. The pool value invariant uses `total_premium − fee_total_sum`, where `fee_total_sum = team_total + partner_totals_sum` by construction.

**Per-policy walker refactor.** The V12 `batch_policies_match_totals_v12` helper (lines 223+) walks outputs once with a `BatchFeeTotals` accumulator. V12.2 keeps the single-walk shape — only the per-policy computation inside the fold changes:

```aiken

# V12 helper (deprecated; kept for cross-version analytics).
def calculate_protocol_fee_split_v12(
    premium_lovelace: int,
    protocol_fee_bps: int,
    partner_share_bps: int,
) -> tuple[int, int]:
    """[v12 DEPRECATED in V12.2] Original V12 split helper.

    Returns (team_cut, partner_cut) such that team + partner == raw_fee.
    V12 callers continue to read this for legacy analytics; new V12.2 code
    paths call calculate_fee_total + calculate_protocol_fee_split.
    """
    total_fee = premium_lovelace * protocol_fee_bps // 10_000
    partner_cut = total_fee * partner_share_bps // 10_000
    team_cut = total_fee - partner_cut
    return team_cut, partner_cut


# V12.2 helpers — Hybrid Fee.
MIN_UTXO_LOVELACE = 2_000_000  # mirror of types.ak::min_utxo_lovelace


def calculate_fee_total(
    premium_lovelace: int,
    protocol_fee_bps: int,
    min_utxo_lovelace: int = MIN_UTXO_LOVELACE,
) -> int:
    """[v12.2] Mirror of Aiken `pricing.calculate_fee_total`.

    Returns max(min_utxo_lovelace, raw_fee). The Aiken validator and this
    helper must produce identical values for every premium / fee_bps tuple,
    or the off-chain build emits txs the chain rejects.
    """
    raw_fee = premium_lovelace * protocol_fee_bps // 10_000
    return max(min_utxo_lovelace, raw_fee)


def calculate_net_pool_growth(
    premium_lovelace: int,
    protocol_fee_bps: int,
    min_utxo_lovelace: int = MIN_UTXO_LOVELACE,
) -> int:
    """[v12.2] Mirror of Aiken `pricing.calculate_net_pool_growth`.

    Returns premium − fee_total. Validator's pool-value invariant on
    Underwrite reads:
      cont_pool.lovelace == own.lovelace + calculate_net_pool_growth(...)
    """
    return premium_lovelace - calculate_fee_total(
        premium_lovelace, protocol_fee_bps, min_utxo_lovelace
    )


def calculate_protocol_fee_split(
    fee_total: int,
    partner_share_bps: int,
    min_utxo_lovelace: int = MIN_UTXO_LOVELACE,
) -> tuple[int, int]:
    """[v12.2] Mirror of Aiken `pricing.calculate_protocol_fee_split`.

    Applies the silent partner_cut absorb rule: sub-floor partner_cut → 0,
    team_cut absorbs the full fee_total.

    Returns (team_cut, partner_cut) such that:
      team_cut + partner_cut == fee_total      (always, exactly)
      partner_cut == 0 OR partner_cut >= min_utxo_lovelace
    """
    partner_cut_raw = fee_total * partner_share_bps // 10_000
    if partner_cut_raw >= min_utxo_lovelace:
        partner_cut = partner_cut_raw
    else:
        partner_cut = 0
    team_cut = fee_total - partner_cut
    return team_cut, partner_cut


def add_protocol_fee_outputs(
    builder: "pyc.TransactionBuilder",
    *,
    premium_lovelace: int,
    protocol_fee_bps: int,
    partner_address: Optional["pyc.Address"],
    partner_share_bps: int,
    network: str = AEGIS_NETWORK,
) -> None:
    """[v12.2] Append team output (always) and partner output (only when
    partner_cut > 0) to the tx builder.

    V12 behavior: emitted floor_pad-padded team and partner outputs,
    always one per address (when partner_share_bps > 0).

    V12.2 behavior:
      * fee_total carries the floor INSIDE the carve.
      * team output threshold = team_cut (no separate pad).
      * partner output OMITTED entirely when partner_cut == 0 (silent absorb).
    """
    import pycardano as pyc
    from chain import (
        AEGIS_PARTNER_SHARE_CAP_BPS,
        get_team_address,
    )

    if partner_share_bps < 0:
        raise ValueError(
            f"partner_share_bps must be >= 0 (got {partner_share_bps})."
        )
    if partner_share_bps > AEGIS_PARTNER_SHARE_CAP_BPS:
        raise ValueError(
            f"partner_share_bps {partner_share_bps} exceeds cap "
            f"{AEGIS_PARTNER_SHARE_CAP_BPS}. The validator rejects this."
        )
    if partner_share_bps > 0 and partner_address is None:
        raise ValueError(
            "partner_share_bps > 0 requires partner_address to be set."
        )

    fee_total = calculate_fee_total(premium_lovelace, protocol_fee_bps)
    team_cut, partner_cut = calculate_protocol_fee_split(
        fee_total, partner_share_bps
    )

    team_address = pyc.Address.from_primitive(get_team_address())

    builder.add_output(
        pyc.TransactionOutput(address=team_address, amount=team_cut)
    )

    if partner_cut > 0 and partner_address is not None:
        builder.add_output(
            pyc.TransactionOutput(
                address=partner_address,
                amount=partner_cut,
            )
        )

def add_batch_protocol_fee_outputs(
    builder: "pyc.TransactionBuilder",
    *,
    policies: list[tuple[int, int, Optional["pyc.Address"], int]],
    protocol_fee_bps: int,
    network: str = AEGIS_NETWORK,
) -> None:
    """[v12.2] Per V12.2 §3.7:
      * Per-policy fee_total_i = max(min_utxo, raw_fee_i) applied individually.
      * Per-policy (team_cut_i, partner_cut_i) via calculate_protocol_fee_split
        (silent absorb on sub-floor partner_cut).
      * team_total = Σ team_cut_i (validator checks single aggregated output).
      * partner_totals[addr] = Σ_{i: partner_i == addr AND partner_cut_i > 0}
        partner_cut_i (silently-absorbed policies don't contribute).
    """
    import pycardano as pyc
    from chain import AEGIS_PARTNER_SHARE_CAP_BPS, get_team_address

    team_total = 0
    partner_totals: dict[bytes, int] = {}
    partner_addr_lookup: dict[bytes, pyc.Address] = {}

        if partner_share_bps < 0 or partner_share_bps > AEGIS_PARTNER_SHARE_CAP_BPS:
            raise ValueError(
                f"per-policy partner_share_bps {partner_share_bps} out of bounds"

        team_cut_i, partner_cut_i = calculate_protocol_fee_split(
            fee_total_i, partner_share_bps

        if partner_cut_i > 0 and partner_addr is not None:
            key = bytes(partner_addr.to_primitive())
            partner_addr_lookup[key] = partner_addr
            partner_totals[key] = partner_totals.get(key, 0) + partner_cut_i

    builder.add_output(
        pyc.TransactionOutput(address=team_address, amount=team_total)

def _resolve_oracle_nft_bytes(provider: str, asset_symbol: str) -> bytes:
    """[v12.2] Network-aware resolver mapping (provider, asset) → oracle NFT.
    
    Used by build endpoints to populate PolicyDatum.oracle_nft at create time.
    The on-chain validator's `is_canonical_oracle_nft` enforces this is one
    of the canonical NFTs for the matching provider.
    """
    if provider == "aegis_self":
        # 5-iAsset map: ADA/BTC/ETH/USDC/USDT. iUSD is NOT here under V12.2.
        return bytes.fromhex(aegis_self.ASSET_TO_NFT_POLICY[asset_symbol.upper()])
    elif provider == "indigo":  # V12.2 NEW
        # 4-iAsset map: IUSD/IBTC/IETH/ISOL. Returns the network-appropriate
        # NFT policy id from chain.py constants.
        from chain import get_indigo_nft
        return bytes.fromhex(get_indigo_nft(asset_symbol.upper()))
    elif provider == "charli3":
        # Soft-disabled in V12.2; the build endpoint should reject before
        # calling here. Defensive fail-closed.

### 5b.2 Mock UTxO shape: 4 controlled UTxOs at operator-owned script

The mock deploys 4 UTxOs at a single operator-owned Plutus V2 script address. Each UTxO carries:

- 1 ADA + min-utxo (~1.7 ADA total).
- A one-shot NFT under a fresh policy minted by the mock-deploy script. Four distinct NFTs (one per iAsset).
- An inline `InlineDatum` with bytes matching Indigo's mainnet wire format exactly: `Constr_0[Constr_0[price_int], expiration_ms]` — bit-for-bit the same shape as the production Indigo OracleDatum.

The script address is a minimal Aiken validator that allows any update (`fn(ctx) { True }` — operator-controlled, no enforcement). This is acceptable because:

- The mock is preprod-only. Mainnet uses Indigo's real validators with `txSignedBy info opOwner`.
- The parser's three-layer trust handshake still applies on preprod: NFT pin (mock NFTs are in `indigo_canonical_nfts`), script-credential pin (mock script hash is in `indigo_oracle_script_hashes`), single-token-of-policy. So a malicious actor on preprod can't forge an iUSD oracle UTxO under their own script — they'd need to mint under the mock NFT policy AND publish at the mock script address. Both are controlled by the operator wallet.
- For mainnet, the parser's same three layers point at Indigo's production NFT policy ids and script hashes — security model intact.

**§10.X (added in this revision) covers the "mock script trust on preprod" risk explicitly.**

### 5b.4 `scripts/refresh_indigo_mock.py PRICE_USD ASSET` — price perturbation

The mock needs perturbation primitives so operator/QA can drive depeg-trigger or crash-trigger paths in preprod testing. The refresh script accepts a price (in USD, decimal) and an iAsset symbol, builds a tx spending the current mock-oracle UTxO and creating a new one at the same script address with the same NFT but the new price + a fresh expiration:

```bash
# Crash iBTC to test claim path for an iBTC-bound policy:
python refresh_indigo_mock.py --asset iBTC --price-usd 200_000.00 --network preprod

# Depeg iUSD to $0.85 to trigger depeg claim:
python refresh_indigo_mock.py --asset iUSD --price-usd 0.85 --network preprod
```

Behavior:

1. Loads `indigo-mock.preprod.json`.
2. Queries the current oracle UTxO at the mock script address carrying the iAsset's NFT (`Koios asset_utxos`).
3. Encodes the new price as 1e6-scaled integer (`int(price_usd * 1_000_000)`) — note that on preprod we publish USD-denominated prices via the mock, because the validator is denomination-agnostic (see §2.6) and mock prices in USD are easier for operator perturbation. The off-chain layer adapts the strike unit accordingly.
4. Builds a tx: consume the current oracle UTxO, mint nothing (NFT is preserved on the continuing output), output back to the mock script address with the new datum `Constr_0[Constr_0[new_price_int], now_ms + 49 min]`.
5. Signs with the operator wallet, submits, prints the new UTxO reference.
6. Updates `indigo-mock.preprod.json` with the new starting UTxO ref.

Estimated tx fee per refresh: ~0.3 ADA. Operator does this manually before each Playwright e2e run that requires a specific price scenario.

### 5b.5 Cost, lifecycle, mainnet cutover deletion

- **Initial deploy cost.** 4 NFT mint txs + 1 script publish tx + 4 oracle UTxO publish txs = ~9 txs × (0.2-0.5 ADA fee + 2 ADA min-utxo) ≈ **15 ADA** sourced from the operator wallet.
- **Ongoing cost.** ~0.3 ADA per refresh; budget ~5 ADA for the V12.2 preprod soak window (assuming < 15 refreshes for full QA coverage).
- **Lifecycle.** The mock lives for the V12.2 preprod lifetime. Until mainnet promotion of V12.2 happens, the mock is the only Indigo binding available on preprod.
- **Mainnet cutover.** When V12.2 is promoted to mainnet, the mock is deleted: operator spends the 4 oracle UTxOs back to the operator wallet (reclaiming the 8 ADA min-utxo), burns the 4 mock NFTs (not strictly required — leaving them out of `indigo_canonical_nfts` mainnet block is sufficient), and the mainnet `indigo_canonical_nfts` block + `indigo_oracle_script_hashes` block (commented out in V12.2; activated at mainnet promotion) point at Indigo's real production values.
- **Why not just keep the mock running on preprod forever.** Drift hazard: if Indigo changes their datum shape on mainnet, the mock would diverge. Mock-deletion at cutover is the forcing function for "preprod must mirror current mainnet shape." If V12.2.x happens (future), Wave 4 reruns with the current Indigo shape baked in.

## 8. Test invariants

V12.2 locks seven property-style invariants. Each is enforced by both an Aiken test and a Python test (round-trip cross-check). Frontend vitest tests assert the same invariants against the off-chain helpers. The auditor verifies each by reading the test and confirming the underlying assertion.

### 8.1 The seven property-style invariants

**Invariant 1 — Premium conservation.**

```
premium == (premium − fee_total) + team_cut + partner_cut
        == net_pool_growth + team_cut + partner_cut
```

For all premiums, all `fee_bps`, all `partner_share_bps`, all `min_utxo_lovelace`. Holds by construction because `team_cut + partner_cut == fee_total` (split helper contract) and `net_pool_growth = premium − fee_total`. The pool growth + team + partner sums to premium exactly.

**Aiken test:**

```aiken
test invariant_1_premium_conservation_holds() {
  let premium = 12_345_678
  let fee_total = calculate_fee_total(premium, 200, 2_000_000)
  let (team, partner) = calculate_protocol_fee_split(fee_total, 1_500, 2_000_000)
  let net = premium - fee_total
  premium == net + team + partner
}
```

**Invariant 2 — Fee floor invariant.**

```
fee_total >= min_utxo_lovelace AND fee_total >= raw_fee
```

The floor is the `max`-form (not `min`-form). For all premiums, `fee_total` clears 2 ADA AND clears the percentage fee. On large premiums `fee_total == raw_fee`; on small premiums `fee_total == min_utxo_lovelace > raw_fee`.

**Invariant 3 — Partner consistency invariant.**

```
partner_cut == 0 OR partner_cut >= min_utxo_lovelace
```

Sub-floor partner cuts are absorbed (→ 0); above-floor partner cuts pass through unchanged. No intermediate "partner_cut between 0 and 2 ADA" state ever exists.

**Invariant 4 — Team output safety.**

```
team_cut >= min_utxo_lovelace
```

Always. Because `team_cut = fee_total − partner_cut`, and `fee_total >= min_utxo` (Inv 2), and `partner_cut ∈ {0, ≥ min_utxo}` (Inv 3), the worst case is `team_cut = fee_total − fee_total = 0` only if `partner_cut == fee_total`. But the validator caps `partner_share_bps <= 2000 = 20%`, so `partner_cut <= 0.2 × fee_total < fee_total`. So `team_cut >= 0.8 × fee_total >= 0.8 × min_utxo >= 1.6 ADA`. When partner is absorbed (`partner_cut == 0`), `team_cut == fee_total >= min_utxo`. Either way `team_cut >= min_utxo_lovelace`.

Hmm, that's not strictly true — `team_cut >= 0.8 × min_utxo == 1.6 ADA` falls short of the 2-ADA floor. Let me re-derive: when `partner_cut > 0`, partner_cut is either the raw value (which equals `fee_total × bps / 10_000`) OR zero. If raw, `team_cut = fee_total × (10_000 − bps) / 10_000`. With `bps <= 2000`, `team_cut >= 0.8 × fee_total`. For `fee_total == min_utxo == 2 ADA`, `team_cut >= 1.6 ADA`, which is **below the min_utxo floor**.

So Invariant 4 actually fails on small premiums with partner-at-max-cap. Concretely: a 100-ADA premium with `partner_share_bps = 2000` → `raw_fee = 2 ADA == min_utxo`, `fee_total = 2 ADA`, `partner_cut_raw = 0.4 ADA < min_utxo` → silent absorb → `partner_cut = 0`, `team_cut = 2 ADA`. The silent absorb saves it — Invariant 4 holds because the absorb activates when partner_cut would have been sub-floor.

Let me re-derive: silent absorb activates iff `partner_cut_raw < min_utxo`. The condition `partner_cut > 0` implies `partner_cut == partner_cut_raw >= min_utxo`. So when partner_cut > 0:
- `partner_cut >= min_utxo`
- `team_cut = fee_total − partner_cut`
- For `team_cut >= min_utxo`, need `fee_total >= 2 × min_utxo`.

The case `fee_total < 2 × min_utxo` requires `raw_fee < 2 × min_utxo = 4 ADA`, i.e., `premium < 200 ADA` at the default fee. Combined with `partner_cut > 0`, this means `partner_cut_raw >= min_utxo` and `fee_total < 2 × min_utxo`, which would require `partner_share_bps × fee_total >= 10_000 × min_utxo`, i.e., `partner_share_bps >= 10_000 × min_utxo / fee_total > 10_000 × min_utxo / (2 × min_utxo) = 5_000`. But `partner_share_bps <= 2000` per the cap, contradiction. So `partner_cut > 0 → fee_total >= 2 × min_utxo → team_cut >= min_utxo`.

Cleaner statement of Invariant 4: **team_cut >= min_utxo_lovelace**, proved by either (a) `partner_cut == 0` → `team_cut == fee_total >= min_utxo`, or (b) `partner_cut > 0` → `partner_share_bps × fee_total >= 10_000 × min_utxo`, with cap `partner_share_bps <= 2000`, forces `fee_total >= 5 × min_utxo`, so `team_cut >= 0.8 × 5 × min_utxo = 4 × min_utxo >= min_utxo`. (Tighter: `team_cut >= 0.8 × fee_total >= 4 × min_utxo`.)

## 10. Risk register

### 10.1 Hybrid fee changes effective rate for small policies

**Risk.** V12.2's Hybrid fee gives slightly less `net_pool_growth` on premiums below 100 ADA — the LP absorbs the floor cost in exchange for a clean user-facing fee curve. This is intentional (§1 design) but warrants explicit comparison.

**Comparison table:**

| Premium | V12 net_premium | V12.2 net_pool_growth | LP delta (V12 → V12.2) | User out V12 | User out V12.2 |
|---|---|---|---|---|---|
| 20 ADA | 19.60 | 18.00 | −1.60 ADA | 22.10 (incl 1.6 pad) | 20.10 |
| 50 ADA | 49.00 | 48.00 | −1.00 | 52.10 (incl 1.0 pad) | 50.25 |
| 80 ADA | 78.40 | 78.00 | −0.40 | 80.80 (incl 0.4 pad) | 80.40 |
| 100 ADA | 98.00 | 98.00 | 0 | 100.50 | 100.50 |
| 200 ADA | 196.00 | 196.00 | 0 | 201.00 | 201.00 |
| 500 ADA | 490.00 | 490.00 | 0 | 502.50 | 502.50 |
| 1000 ADA | 980.00 | 980.00 | 0 | 1005.00 | 1005.00 |

LP gives back ≤1.6 ADA per small-premium policy. The total economic exposure is bounded: a flood of 1-ADA-premium policies (worst case) would pay ~0 net pool growth, but the `min_premium` constant (2 ADA) gates the lower bound. The maximum LP loss per policy is `min_utxo_lovelace − min_premium × fee_bps / 10_000` = `2 ADA − 0.04 ADA` = `1.96 ADA`. For a 100-policy preprod test batch, max LP delta is 196 ADA — well within operator wallet budget for the smoke test.

**User-visible fee at 20 ADA.** V12 effective rate: `(2.50) / 20 = 12.5%` (including the 1.6 ADA floor pad). V12.2 effective rate: `0.10 / 20 = 0.5%` (excluding any pad — but the "fee" the user feels is just the treasury donation, because the pool absorbed the team fee). Both versions extract the same gross fees to the protocol; V12.2's UX just moves the visible cost from the user's wallet to the pool's net growth.

**Mitigation.** Set `min_premium` to a value high enough that the LP absorption is bounded. Currently `min_premium = 2 ADA` per `types.ak:284` — a 2-ADA premium gives `raw_fee = 0.04 ADA`, `fee_total = 2 ADA`, `net_pool_growth = 0`. The pool gets 0 lovelace for a 2-ADA insurance policy. Operator could raise `min_premium` to 10 or 20 ADA to ensure the LP always gets at least some lovelace. **Recommended: keep min_premium at 2 ADA for V12.2** (mainnet may revisit). The economic exposure is small relative to the simplification gain.

### 10.2 Soft-disable leak surface

**Risk.** A code path that bypasses `is_canonical_oracle_nft` could let an attacker create a V12.2 policy with `oracle_provider: Charli3` and a hand-crafted `oracle_nft`. The validator would then attempt to resolve via `oracle/charli3.ak::resolve` (which is dead code with leading `fail`).

**Code path audit checklist.** The Wave 2 agent walks every entry point that touches `PolicyDatum.oracle_provider` and confirms `is_canonical_oracle_nft` is called:

1. **`validators/pool.ak` Underwrite branch.** Calls `policy_output_matches_underwrite` (line 654 at HEAD `751c00c`), which (in V12) verifies the canonical-handle pin inside the helper. Wave 2 verifies the V12.2 walker preserves this check. The Underwrite branch CANNOT create a policy with a non-canonical oracle pin. ✓
2. **`validators/pool.ak` BatchUnderwrite branch.** Calls `batch_policies_match_totals_v12_2` (V12.2 walker, §3.7), which verifies per-policy canonical handle inside the fold. ✓
3. **`validators/pool.ak` ProcessClaim branch.** Reads `PolicyDatum.oracle_provider` from consumed policy input. The consumed policy was created via Underwrite or BatchUnderwrite (which gated it). No claim-time canonical-handle check needed — the binding is already pinned. ✓
4. **`validators/pool.ak` AcceptCancellation branch.** Same shape as ProcessClaim — consumed policy already gated at Underwrite. ✓
5. **`validators/pool.ak` BatchExpireProcess branch.** Same. ✓
6. **`validators/policy.ak` Claim / Cancel / Expire / BatchClaim / BatchExpire.** Off-chain orchestration only; the policy-side validator doesn't read `oracle_provider` (the resolver is called by the pool-side via `resolve_oracle_price`). Wave 2 confirms via source read. ✓

If any of these surfaces fail the audit, the `fail @"v12.2: ..."` in the resolver bodies is the second layer of defense — even an unvetted reach to `oracle/charli3.ak::resolve` aborts the tx.

**Mitigation.** The audit checklist above is the only required defense beyond the canonical-handle pin. The `fail` is belt-and-braces.

### 10.3 Silent partner_cut absorb security analysis

**Risk.** When `partner_cut_raw < min_utxo_lovelace`, the validator omits the partner output and `team_cut` absorbs the full `fee_total`. PolicyDatum still records `partner_address` and `partner_share_bps`. Could an attacker exploit this?

**Attack surface enumeration:**

1. **Diverting team_cut to a fake "partner".** The attacker sets `partner_address = <attacker_address>`, `partner_share_bps = 1500`. On a 500-ADA premium, `partner_cut_raw = 1.5 ADA < 2 ADA` → absorbed. team gets full 10 ADA. The attacker receives 0 ADA — they self-pwned (intended their address to get the cut). **Not an attack.**

2. **Pool drain via silent absorb.** The validator's pool value invariant is `cont == own + (premium − fee_total)`. The `fee_total` is fixed by the premium and the fee_bps; it doesn't depend on partner_share_bps. So the pool's value reduction is identical regardless of whether the partner_cut is absorbed or paid. **No drain surface.**

3. **Team address compromise.** team_address is compile-time pinned (`types.ak:545` per V12 §3.5). Even if silent absorb routes 100% of fee_total to team_cut, the team output lands at the pinned address — operator-controlled (or Flux Point Studios on mainnet). An attacker can't redirect via this path.

4. **Off-chain bookkeeping inconsistency.** The partner registry reads `PolicyDatum.partner_address` + `partner_share_bps` for "this policy belongs to partner X with Y% share." If the validator absorbed the partner_cut, the registry's "Y% of fees attributed to partner X" claim is inaccurate. **Mitigation:** the partner registry's analytics MUST cross-check the actual `partner_cut` value by computing it from premium + fee_bps + partner_share_bps + min_utxo at the off-chain layer, NOT by trusting the datum's bps directly. The registry's "earnings" calculation factors in absorb: `partner_earnings = sum over policies of max(0, partner_cut_for_that_policy)`. This is documented in the partner registry's API docs at `docs/api-reference.md` (Wave 5 agent writes the cross-check helper).

5. **Race condition between off-chain calc and validator absorb.** The off-chain `add_protocol_fee_outputs` helper (§4.2) MUST compute the absorb rule identically to the validator. A divergence would emit a tx with a partner output the validator rejects. **Mitigation:** the Python `calculate_protocol_fee_split` helper mirrors the Aiken function bit-for-bit. Pytest cross-checks against canonical fixtures (§8.3) ensure no drift.

**Conclusion.** No attack surface. The silent absorb is a UX simplification with zero security implication. team_address is pinned, pool delta is fixed by premium, fee_total is deterministic from premium + fee_bps + min_utxo, and the partner registry handles the bookkeeping cleanly off-chain.

### 10.4 Indigo keeper-liveness dependency (was: "iUSD publisher dependency on Indigo Analytics API uptime")

**Risk.** Indigo's keeper VKHs (one per iAsset) are the entities that refresh each iAsset's oracle UTxO via the `FeedPrice` redeemer. If a keeper stalls, the on-chain UTxO's `odExpiration` runs out (typically ~49 min after the last publish), and all Aegis policies bound to that iAsset become unclaimable until the next refresh (the validator's freshness check `tx_lower <= datum.expiration` rejects).

**Mitigation:**

1. **Aegis claim-time freshness gate.** The validator already enforces `tx_lower_bound <= datum.expiration`. A stale Indigo oracle UTxO produces claim-tx rejection — no bad payouts emit. The user's policy survives the stall (it doesn't expire from Aegis's side; only the claim-eligibility window is paused). When Indigo's keeper resumes refreshing, claims become eligible again.

2. **Cadence empirical** (Wave 1.5 measurement, mainnet iUSD oracle, last 1000 txs):
   - Mean: 35.5 min
   - Median: 40 min
   - Min: 5.3 min
   - Max: 132 min (occasional 2h+ gap)

   The 132-min max means a worst-case keeper stall is ~2h on observed history; the validator's freshness gate is `tx_lower <= expiration` where expiration = publish + ~49 min, so the gap between publish and claim-eligibility-expiry is ~49 min. A stall longer than 49 min means temporarily unclaimable. Documented in the Aegis frontend as "occasional 2h+ keeper delays observed historically; claims may briefly pause."

3. **No operator action required.** Unlike a publisher outage (V12.1 AegisSelf path), an Indigo keeper outage is NOT something the operator can fix. The operator just waits. The frontend displays a "Oracle refresh pending" badge on Indigo-bound policies when the on-chain datum's expiration is < 5 min away or already past.

4. **Multi-provider redundant binding** (future work, §13). If Indigo's keepers stall persistently, a policy could be re-bound to AegisSelf (if we publish that iAsset) via an off-chain rebind. Not in V12.2 scope.

5. **Cross-protocol stress correlation.** During Cardano-wide market stress (e.g., CRH cascade), Indigo's keepers may delay AND iAsset markets may decouple from real-world prices. This is captured in §10.7 (basis risk).

**Operational consequence.** Indigo-bound policies are slightly less always-claimable than AegisSelf-bound policies (which we control). This trade-off is acceptable for V12.2 — the upside (4 new iAssets covered without publisher work) outweighs the occasional claim-window pause. Documented in the auditor checklist (§11.7).

### 10.4b Mock-script trust on preprod (NEW for V12.2 Indigo mock)

**Risk.** The preprod Indigo mock (§5b) deploys 4 oracle UTxOs at an operator-owned trivial-validator script address. Any holder of the operator's wallet can spend those UTxOs and set arbitrary prices/expirations. There is NO trust handshake beyond "operator controls the mock script."

**Why this is acceptable:**

1. **Preprod-only.** The mock script hash is in the preprod block of `indigo_oracle_script_hashes`. The mainnet block points at Indigo's real per-iAsset script hashes. So an attack on the preprod mock has zero implication for mainnet policies.

2. **Operator-controlled.** The mock-deploy script's mnemonic is the operator's own (same wallet that runs the AegisSelf publishers, deploys ref scripts, etc.). An attacker would need to compromise the operator's hot wallet to perturb preprod mock prices — at which point the rest of preprod is also compromised.

3. **Preprod policies are operator test policies.** Real users do NOT buy V12.2 preprod policies (the staging frontend is operator-facing, gated behind the Vercel preview URL). So even if the operator wallet were compromised and mock prices flipped to trigger fake claims, no real value is lost.

4. **Mainnet has the real Indigo handshake.** The validator's three-layer trust handshake (NFT pin + script pin + single-token-of-policy) is identical on preprod and mainnet — only the canonical NFT + script hash constants differ. On mainnet, those constants point at Indigo's real production values, which means the keeper VKH transitively signs every refresh (Indigo's own validator enforces). The mock-script trust gap exists ONLY on preprod by design.

**Mitigation that's explicitly NOT required:**

- We do NOT need to enforce keeper-VKH signatures on the preprod mock (Indigo's mainnet does this; the mock is for shape testing only).
- We do NOT need to require the mock script address to be unique per iAsset (mainnet uses 4 different addresses due to Indigo's parameterization; preprod uses 1 shared address to reduce dev-loop cost, see §5b.2).

**Documentation requirement.** The mock-deploy script's stdout output, the V12.2 spec, and the V12.2 preprod runbook all carry a "PREPROD ONLY — operator-controlled mock, NO TRUST HANDSHAKE" warning. Wave 4 confirms this is visible in the script's output banner before exiting.

### 10.4c Real BTC vs iBTC basis risk (NEW for V12.2 Indigo synthetic assets)

**Risk.** Aegis covers **oracle price**, not market price. A user who buys an iBTC crash policy via the Indigo binding is insuring against Indigo's published iBTC price (which is BTC/USD with a CDP-stress-aware adjustment, plus the Cardano-DEX iBTC trading price drifting from real BTC). The two prices can diverge under Cardano-specific stress (e.g., iBTC peg breaks during a stability-pool liquidation cascade; real BTC unchanged on Coinbase but iBTC drops 20% on Minswap).

**Implications:**

1. **A user who wants real-BTC coverage should buy ADA/BTC/ETH** (the Real-asset selector group, AegisSelf-bound). Those track CEX-aggregated USD price.
2. **A user who wants synthetic-stress coverage should buy iBTC/iETH/iSOL** (the Synthetic-Indigo group). Those track Indigo's published price (i.e., insure against Indigo CDP system stress / synthetic-asset decoupling).
3. **The frontend UI must signal this difference.** The two-section BuyPanel selector with tooltips (§6.2) is the primary affordance; the tooltip text explicitly mentions "Synthetic = Cardano-native, can decouple from real-world price under Cardano-specific stress."

**No on-chain mitigation needed.** The validator is denomination-agnostic and just compares oracle to strike. The risk lives in the user's mental model — captured in §11.5 auditor checklist as "BuyPanel tooltip prose must be reviewed for completeness."

**Documentation requirement.** The Aegis docs site adds a FAQ entry "What's the difference between buying ADA/BTC/ETH coverage vs iBTC/iETH/iSOL coverage?" with a 2-paragraph explanation.

### 10.5 Single-redeploy risk vs phased

**Risk.** V12.2 bundles four changes (Hybrid fee, soft-disable, β-stage iUSD, silent absorb) into one redeploy. If any one of the four breaks, the entire deploy rolls back to V12. A phased rollout (Hybrid fee first, then soft-disable, then iUSD, then absorb) would isolate failures.

**Why we don't phase:**

1. **All four touch `lib/aegis/types.ak` or `lib/aegis/pricing.ak` or `validators/pool.ak`.** Any single change rotates all three application validator hashes. So a phased rollout would be three separate deploys, each rotating hashes, each requiring full env-var rotation + ref-script publishing + auditor pass.

2. **Auditor cost.** Each deploy costs ~$10-20K in auditor fees (external auditor turnaround). Three deploys × auditor = $30-60K vs one deploy × auditor = $10-20K. The total review surface is the same — the auditor reviews the same diff either way; bundling just amortizes the per-deploy overhead.

3. **V11→V12 stranded-policy story.** Each phased deploy creates a new V12.x version with its own stranded policies. Bundling avoids playing the V11 sunset story out four times in a week.

4. **Coordination cost.** Four phased deploys means coordinating four operator manual actions (Wave 6 cutover), four sets of preprod soak tests, four operator runbook entries. One deploy = one of each.

**Risk mitigation:**

- **Aiken test surface.** 7 invariants × multiple positive/negative tests = comprehensive validator-side coverage. The Wave 2 agent writes ~30 new tests (10 V12.2-specific + 20 V12-compat regression). All must pass before Wave 3 starts.

- **Pytest cross-check.** §8.3 fixtures pin canonical fee math against Aiken values. Pytest CI runs on Railway; any divergence between Python and Aiken trips immediately.

- **Vitest cross-check.** §8.4 fixtures pin frontend math against Python (and transitively Aiken).

- **Playwright smoke.** §8.5 e2e tests run against staging URL post-Wave 5 deploy. A bad fee-math tx would fail on submit and trip the test before any user ever sees it.

- **Rollback.** Railway/Vercel env-var flip back to V12 reverses the deploy in < 5 min. The V12 validator hashes stay published — no ref-script tomb-stoning. The dApp resumes pointing at V12; only the V12.2 policies created during the V12.2 window become stranded (per §9.2 pattern, claimable via CLI against V12.2 hash).

### 10.6 Rationale: soft over hard disable

**Risk consideration.** A hard-disable (removing `Charli3` and `Orcfax` variants from `OracleProvider`) would simplify the type at the cost of breaking V11 datum decode. A soft-disable (return False in canonical check) keeps the variants but blocks runtime use. Which is safer?

**Why soft.**

1. **V11 policy decoding.** V11 policies on preprod (4 stranded) carry `oracle_provider: Charli3`. If V12.2 validator code paths anywhere (cross-validator helpers, off-chain analytics) needed to decode a V11 datum, a hard-disable would fail the decode. Soft-disable preserves decoder symmetry.

2. **Reactivation cost.** If Charli3 stabilizes or a community Orcfax fork emerges, reactivation under soft-disable is a one-line `is_canonical_oracle_nft` change. Under hard-disable, it's a schema rotation (add variant back) which is a much larger audit surface.

3. **Type-system safety.** Soft-disable keeps Aiken's exhaustivity check on `when provider is { ... }`. A future variant addition still requires explicit branch coverage. Hard-disable would shrink the type, and a future re-expansion would re-introduce the same shape — net-zero in safety, net-positive in operational cost.

**Why NOT hard.** The only argument for hard is "smaller type, less surface to audit." But the auditor reviews all three arms under either approach (because the soft-disable change is a 2-line diff in `oracle.ak`, and the hard-disable change is a multi-line `OracleProvider` enum + datum-schema rotation). Soft has equal audit surface and better operational ergonomics.

**Decision: soft.** Locked.

## 13. Inconsistencies / notes from grounding

Flagged for the operator. None are blocking — notes from reading the V12 spec + V12 source code against the V12.2 brief.

- **The user brief shows `partner_cut_raw = fee_total × partner_share_bps / 10_000`, with `partner_share_bps` being bps OF THE FEE.** V12 spec §3.5 / §3.7 also treated `partner_share_bps` as bps OF THE FEE (not of the premium), at the validator level. V12.2 preserves this convention. Section 1's worked example "500 ADA, S=1500" produces `partner_cut_raw = 10 × 1500 / 10_000 = 1.5 ADA`, confirming the interpretation. The 20% cap at `partner_share_cap_bps = 2000` thus means "up to 20% of the protocol fee" = "up to 0.4% of the premium" — same numbers as V12.

- **V12's `floor_to_min_utxo` helper in `pool.ak`** (lines 479-485 at HEAD `751c00c`) becomes unreachable in V12.2 active validator branches. Wave 2 has two options: (a) delete the helper entirely (cleaner), (b) leave in tree marked `// V12 compat helper — unreachable in V12.2`. Recommended: **delete** if no V12-compat code path needs it; the function is a 6-line helper and removing it shrinks the audit surface. If any helper outside the three V12.2 active branches still calls it, retain.

- **V12's `output_to_address_at_least` helper** (lines 497-508) stays — V12.2 uses it unchanged.

- **`calculate_treasury_cut`** (V12 `pricing.ak:76`) is untouched in V12.2. The treasury math is asset-agnostic to the Hybrid fee floor (it computes against `premium × fee_bps × treasury_share_bps / 1e8`, independent of `fee_total`). This means a 20-ADA premium pays 0.10 ADA treasury both under V12 and V12.2.

- **V11→V12 stranded policies (4 preprod policies, ~97 ADA premium)** persist into V12.2 as the "V11→V12 sunset story" — V12 left them stranded at V11 hash; V12.2 leaves them in the same state. No additional action needed in V12.2 to handle V11; the V11 CLI escape hatch from V12 spec §9.1 still works.

- **The `Aiken @deprecated` attribute** doesn't exist in Aiken v1.1.21. The deprecation marker for `oracle/charli3.ak` + `oracle/orcfax.ak` is therefore a doc comment + a leading `fail` in the `resolve` function. Wave 2 verifies the `fail` syntax compiles in this position; if Aiken requires the trailing body to be removed, the body moves to a parallel `resolve_dormant` function (signature preserved for any tooling that reflects on the file). Spec records both paths for Wave 2.

- **Indigo NFT + script-hash placeholders.** (Replaces the cancelled Wave-1 "iUSD NFT hash placeholder" entry.) The preprod Indigo binding carries placeholders in **two** files (after Wave 1.5 pivot): `types.ak` (4 NFT entries in `indigo_canonical_nfts` + 4 entries in `indigo_oracle_script_hashes`) and `api/chain.py` (8 `AEGIS_INDIGO_*` constants for preprod, plus 4 mainnet asset name constants). Wave 4 substitutes all placeholders ATOMICALLY in a single commit after `scripts/init_indigo_mock.py` runs. A partial substitution emits a compile error in Aiken or fails the `pytest -q` cross-check.

- **BatchFeeTotals record collapse.** V12's `BatchFeeTotals` has 8 fields; V12.2's has 7 (`partner_totals_unfloored_sum` removed; `team_total_floored` and `team_total_unfloored` collapsed to one `team_total`; `partner_totals_floored` becomes `partner_totals`). The collapse is enabled by the silent absorb rule's invariant (`team_cut + partner_cut == fee_total` exactly; no floored/unfloored distinction needed). Wave 2 confirms no V12 call site outside `pool.ak` depends on the dropped fields.

- **PolicyDatum field count.** V12 has 13 fields. V12.2 does not change the field count — the new Indigo-iAsset wire-up happens via the existing `oracle_provider: OracleProvider` + `oracle_nft: ByteArray` fields (provider value is now one of 4 variants instead of 3; oracle_nft is now a member of one of the per-provider canonical lists), not via new datum fields.

- **`min_utxo_lovelace = 2_000_000` constant.** `types.ak:299`. V12.2 passes this into `calculate_fee_total` and `calculate_protocol_fee_split` as an explicit argument rather than hardcoded in each helper. This makes the helpers more testable (vary the floor in tests) and keeps the single source of truth in `types.ak`.

- **Indigo Analytics API rate limits — moot post-pivot.** The Wave-1 plan relied on Indigo's REST API at 1 fetch/60s; the pivoted V12.2 plan does NOT consume the REST API for any validator-bound path (we read on-chain UTxOs directly via Koios or Blockfrost). So Indigo's API rate limits are no longer in scope for V12.2. The unrelated `api/lending/indigo.py` CDP-browse path still uses the REST API; its rate-budget concern is unchanged from V12.

- **Indigo's iSOL canonical asset name shape divergence.** Three of Indigo's iAsset NFTs have asset names like `iETH20221219191302` (= symbol + UTC timestamp at mint, hex-encoded ASCII) — but iSOL's asset name is `iSOL_ORACLE` (= literal string, no timestamp). The parser pins the (policy_id, asset_name) tuple per-iAsset via `chain.py::AEGIS_INDIGO_*_NFT_ASSET_NAME_MAINNET`. The iSOL divergence is a fact of Indigo's deployment history; our parser handles it correctly because we pin both fields explicitly. No on-chain code path depends on the asset name following the timestamp pattern.

- **Indigo's IAssetDatum vs OracleDatum confusion.** The Indigo Analytics REST API's `/api/v1/assets/` endpoint returns 4 entries with `outputHash` + `outputNftCs`. The `outputHash` points at the **IAssetDatum** UTxO (at the CDP-script address `addr1wyyqtkz5rken7jzptp076np606r79lmsrqjrqw8sdn4kvrqewrkdg`) which is the iAsset CDP registry — NOT the price oracle. ### 15.2 Multi-provider redundant binding (V13 candidate)

Today a policy's `PolicyDatum.oracle_provider` and `oracle_nft` are frozen at Underwrite time. If Indigo's keepers stall persistently on a specific iAsset (e.g., iSOL keeper goes silent for 24h), all live iSOL-bound policies become temporarily unclaimable.

**Future redesign.** Add a `secondary_oracle_provider` + `secondary_oracle_nft` pair to `PolicyDatum`, with a tie-breaking rule: claim-time validator uses primary if fresh, falls back to secondary if primary's `valid_until` has expired. Off-chain rebind script (operator-driven) writes the secondary fields after policy creation. Requires:

- PolicyDatum schema extension (13 → 15 fields). Validator-hash rotation.
- Off-chain rebind logic in `api/policies.py`.
- Frontend UI to display "primary: Indigo iBTC | secondary: AegisSelf BTC (fallback only)" badges.

The architectural option is captured for when Aegis itself publishes the 4 Indigo iAssets via AegisSelf (a V13+ effort). Until AegisSelf has equivalents to iBTC/iETH/iSOL — which would require Indigo-CDP-aware price composition logic that Aegis doesn't run today — multi-provider redundant binding has no second source to bind to. V12.2's mainnet target is single-provider per policy.
